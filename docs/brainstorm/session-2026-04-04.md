## Refactor Thesis

ViePilot v2 đang mang nhiều giá trị thực tế nhưng bị vá chồng từ nhiều ý tưởng và nhiều lớp tài liệu. Pain cốt lõi không nằm ở việc thiếu capability, mà ở chỗ runtime phải đọc quá nhiều prose để suy ra operational truth, dẫn đến token waste, docs drift, state sync burden, và trải nghiệm xấu hơn ở các host dễ bị giới hạn như Claude Code và Cursor.

V3 cần được thiết kế lại như một **state-machine-first framework** với **compiler-driven runtime**:
- brainstorming vẫn giữ richness
- extraction phải strict
- compile phải tạo runtime truth
- projections không còn là source of truth

## Topics Discussed

### Topic 1: Core pain statement

**User goal**:
- Giữ nguyên flow hoạt động hiện tại
- Tổ chức lại toàn bộ framework để giảm token costs, gaps, và docs duplication
- Tăng hiệu quả + trải nghiệm khi dùng với Claude Code, Cursor, Codex

**Primary pain groups confirmed**:
- `Context/docs`: cold start nặng, nhiều file phải đọc, prose duplication
- `Workflow structure`: nhiều bước và rule đang sống trong narrative docs
- `State persistence/artifacts`: quá nhiều artifact cùng mang runtime meaning

**Extra pain points surfaced during deep-dive**:
- Narrative duplication giữa skill/workflow/docs/tracker/task/changelog
- Runtime truth bị phân tán
- Resume path quá semantic, phải hiểu lại câu chuyện
- Compatibility đang là prompt burden hơn là adapter layer thực
- Refactor history layered patches làm workflow ngày càng exception-heavy

**Decisions**:
- [x] v3 phải tối ưu trước cho **Claude Code + Cursor**
- [x] token waste là vấn đề kiến trúc, không chỉ là vấn đề rút ngắn tài liệu

---

### Topic 2: High-level architecture direction

**Direction selected**:
- `Fresh v3 architecture`
- `State machine framework first`
- `One source of truth`
- `Compiler-driven runtime`
- `One-shot migration`

**Chosen architecture**:
- `State Core + Compiler`

**Implication**:
- planning artifacts phục vụ planning/human review
- compiler tạo runtime artifacts
- runtime chỉ đọc structured artifacts

**Decisions**:
- [x] v3 lấy **state machine** làm lõi trước
- [x] source of truth duy nhất là structured core, không phải prose markdown
- [x] migration là **one-shot**, không kéo dài dual-runtime

**Decisions**:

- [x] projection artifacts chỉ còn human/compliance/compatibility value

### Topic 4: Runtime read policy

**Chosen policy**:
- `/vp-auto` runtime read policy: `Never`

**Meaning**:
- `/vp-auto` chỉ đọc:
  - `execution-graph.json`
  - `runtime-state.json`
  - `active-packet.json`
- Nếu thiếu context ở runtime thì lỗi nằm ở compiler/packet design, không sửa bằng prose fallback

**Decisions**:
- [x] `/vp-auto` không được đọc planning prose để vận hành trong v3
- [x] packet/compiler quality là nơi phải được đầu tư để giảm token waste thật sự

### Topic 5: Flow redesign map

**`/vp-request`**:
- Chosen role: **adaptive front door**
- Policy: **hybrid with memory**
- Must handle fuzzy user intent và tự route theo context project + workflow history + confidence

**`/vp-brainstorm`**:
- Chosen role: **rich discovery workspace**
- Không tách research / imagination / exploration ra khỏi core strength của brainstorm
- Pain nằm ở extraction boundary, không nằm ở brainstorm richness

**`brainstorm extraction`**:
- Chosen model: **hybrid**
- Session vẫn giàu nhưng có structured anchors nhẹ
- Sau đó chạy extraction pass để normalize + validate cho compiler/planning layer

**`/vp-crystallize`**:
- Chosen role: **two-stage**
- Stage A: extract + normalize
- Stage B: compile + project

**`/vp-evolve`**:
- Chosen role: **planning workspace**
- Sửa planning intent, tạo delta, preview impact, rồi recompile

**`/vp-auto`**:
- Chosen role: **runtime executor only**
- Không còn đóng vai interpreter của planning prose

**Decisions**:
- [x] `/vp-request` không chỉ là backlog logger nữa
- [x] `/vp-brainstorm` phải giữ richness
- [x] extraction phải strict hơn brainstorm
- [x] `/vp-crystallize` và `/vp-evolve` trở thành planning/compiler-oriented

## Summary

### Final Thesis
- ViePilot v3 nên là một **state-machine-first, compiler-driven runtime framework**
- `/vp-brainstorm` giữ nguyên sức mạnh discovery workspace
- `brainstorm extraction` phải trở thành boundary chặt chẽ giữa creative discovery và planning/compiler layer
- `/vp-auto` trong v3 không đọc planning prose nữa
- state truth phải tập trung vào structured runtime artifacts; markdown chỉ còn là projection/view/human interface

### Why this direction
- giảm token waste tận gốc
- giảm docs drift
- giảm state sync burden
- cải thiện resume/control-point continuity cho Claude Code + Cursor
- tránh tiếp tục vá thêm trên kiến trúc v2 đã layered patches
