# ENH: crystallize UI→Task Binding — prototype pages must map to implementation tasks

## Meta
- **ID**: ENH-069
- **Type**: Enhancement
- **Status**: done (Phase 103 → v2.38.0)
- **Priority**: high
- **Created**: 2026-04-22
- **Reporter**: User
- **Assignee**: AI

## Summary
After running all crystallize-generated phases, UI Direction prototype pages (e.g. `landing.html`) remain unimplemented as stubs. The root cause is a **gap chain** in the crystallize workflow: it reads `pages/*.html` and understands them, but never creates a mandatory binding from "prototype file" → "implementation task". Tasks are generated from `phases_inventory` (brainstorm Q&A scope), which may not include every designed page.

## Root Cause Analysis — Gap Chain

```
brainstorm --ui
  → landing.html created (full prototype)
  → phases_inventory written from Q&A scope (auth, songs, search, presentation)
  
crystallize Step 1A
  → reads pages/*.html ✅
  → extracts layout/components ✅
  → does NOT create a "pages → tasks" binding document ❌
  
crystallize Step 7 (Generate ROADMAP.md)
  → generates phases from phases_inventory
  → phases_inventory had auth/songs/search priority
  → landing page NOT in phases_inventory → NO task generated for it ❌
  → no enforcement that every page/*.html has a corresponding task ❌
  
vp-auto Phase 1
  → creates Home.vue as minimal scaffold for "Inertia routing works"
  → task file has no ## UI Prototype Reference field ❌
  → task never said "implement from landing.html" ❌
  
Phase 3 complete
  → landing.html prototype exists at .viepilot/ui-direction/session-*/pages/landing.html
  → Home.vue is still a 2-line stub ❌
  → No phase completion gate checked UI coverage ❌
```

## Ten Specific Gaps (full research)

### Gap 1 — crystallize Step 1A: reads pages/*.html but produces no binding
**Current:** Step 1A reads each `pages/{slug}.html` for "section structure, components, and interaction hints" — but this information is only fed as context to the LLM. There is no output artifact that maps `pages/landing.html → must implement resources/js/Pages/Home.vue`.

**Fix:** After reading pages/*.html, emit a `## UI Pages → Component Map` table to working notes:
```
| Prototype | Target component | Phase | Status |
|-----------|-----------------|-------|--------|
| pages/landing.html | resources/js/Pages/Home.vue | Phase 1 | pending |
| pages/dashboard.html | resources/js/Pages/Dashboard.vue | Phase 2 | pending |
```

### Gap 2 — crystallize Step 7: ROADMAP generated from phases_inventory only
**Current:** Step 7 generates ROADMAP from `phases_inventory` without cross-checking against the UI Pages → Component Map. Pages designed in UI Direction mode but not mentioned in phases_inventory Q&A are silently dropped.

**Fix:** After generating ROADMAP from phases_inventory, add a mandatory step: "For each row in UI Pages → Component Map — verify a task exists in ROADMAP that implements it. If missing → add a task to the appropriate phase."

### Gap 3 — vp-auto task files: no `## UI Prototype Reference` field
**Current:** Task files generated by crystallize/vp-auto have no field pointing to the relevant prototype HTML file. A task like "Create Home.vue" has no indication that `landing.html` should be used as the design source.

**Fix:** Add `## UI Prototype Reference` field to task template for frontend component tasks:
```markdown
## UI Prototype Reference
- `.viepilot/ui-direction/{session-id}/pages/landing.html` — use as implementation source
- Key sections: hero banner, search bar, feature list, CTA footer
```

### Gap 4 — No UI coverage gate at phase/session completion
**Current:** No check at phase end or `/end` session that all ui-direction pages have been implemented (not stub). The project can "complete" Phase 3 with placeholder components.

**Fix:** Add a UI Coverage Gate — before marking a phase PASS or running `/end`:
- Check the UI Pages → Component Map
- For each pending row whose Phase ≤ current phase: verify target component file is NOT a stub (heuristic: file exists AND length > N lines or specific content check)
- If stubs found: warn user and list unimplemented pages before allowing PASS

### Gap 5 — crystallize Step 1D: arch_to_ui_sync `status: noted` items silently dropped
**Source:** `brainstorm.md` `arch_to_ui_sync` records UI implications from architectural decisions. Items with `status: noted` mean "we detected this during brainstorm but didn't address it in UI Direction." These are stored in `notes.md ## arch_to_ui_sync`.

**Current:** crystallize Step 1A and Step 1D both list `arch_to_ui_sync[]` as something to read (lines 995, 1107) — but there are **zero instructions on what to do with them**. `status: noted` items representing unaddressed UI screens/states are read then silently dropped.

**Fix:** In crystallize Step 1D (after reading notes.md), for each `arch_to_ui_sync` entry with `status: noted`:
- Check if the referenced `ui_implication` (screen/state) exists in `pages/*.html`
- If NOT: add an entry to the UI Pages → Component Map (Gap 1 fix) with source `arch_to_ui_sync`
- If YES: no action needed (already covered by page binding)

### Gap 6 — crystallize Step 1F: coverage[] gaps are non-blocking warnings only
**Source:** `notes.md ## coverage` matrix (from brainstorm Architect mode) tracks which features have architect/UI coverage. Step 1F (Cross-Reference Gate) warns about features with "none yet" in both columns — but this is **non-blocking** (can be dismissed).

**Current:** `⚠️ Coverage gap: Feature "{name}" has no coverage in Architect OR UI Direction.` — user can dismiss and proceed without creating a task for it.

**Fix:** For coverage gaps where the feature is in `phases_inventory` (i.e. should be implemented), make the warning **blocking**: "This feature is scoped to a phase but has no design coverage. Options: (a) add to UI Direction now, (b) create implementation task with explicit 'design-TBD' note, (c) remove from scope." Cannot silently skip a scoped feature with zero design coverage.

### Gap 7 — crystallize Step 1D item 11: feature-map.html discrepancies are "list only"
**Source:** `feature-map.html` shows features with Phase/priority/status badges. Step 1D item 11 says: "cross-reference Phase badges with `phases_inventory`; if discrepancies found (feature in HTML not in inventory, or vice versa) → **list them for user to confirm**."

**Current:** Listing for confirmation is not the same as resolution. A feature tagged Phase:1 in feature-map.html but absent from phases_inventory gets a list entry that the user may dismiss — then ROADMAP Phase 1 silently omits that feature.

**Fix:** Discrepancies must be explicitly resolved before ROADMAP generation: each discrepancy gets one of: (a) add to phases_inventory and ROADMAP, (b) mark feature as removed/descoped with reason, (c) flag as design-only (no implementation task). The response is logged in working notes before Step 7 runs.

### Gap 9 — crystallize Step 1A: `## UX walkthrough log` consumed but never actioned
**Source:** `/research-ui` command (FEAT-010 + ENH-019 + ENH-020) writes `## UX walkthrough log` to ui-direction `notes.md`. This section contains: scenario walkthroughs, Stress findings (content stress + archetype recipes), and **P0/P1/P2 pain priority items** with design root causes.

**Current:** crystallize Step 1A line 995 says read "ALL sections" of notes.md including `ui_idea_buffer[]`, `background extracted ideas`, etc. — but `## UX walkthrough log` is not even mentioned in that list, and there are **zero instructions anywhere in crystallize.md** on what to do with P0/P1/P2 pain findings. Pain items with real user impact (content loss, mis-clicks, unusability) are silently read and dropped.

**Fix:** After reading `notes.md`, for each entry in `## UX walkthrough log` with severity `P0` or `P1`:
- Add to the UI Pages → Component Map (Gap 1 fix) as a UX-fix row with source `ux_walkthrough`
- Generate a task `fix-ux: {pain description} on {page}.html` in the appropriate phase
- P2 items: surface as warnings only (non-blocking)

### Gap 10 — crystallize Step 1A: `## Background extracted ideas` silently dropped
**Source:** When brainstorm's Background UI Extraction (ENH-026) fires the confirmation dialogue and user selects **Option 1 (notes only, no HTML)**, ideas are saved as `## Background extracted ideas` in notes.md. These represent UI scope the user explicitly acknowledged but chose not to build a prototype for immediately.

**Current:** crystallize Step 1A lists `background extracted ideas` as a mandatory read (line 995) — but there are **zero follow-up instructions** in crystallize.md on what to do with these ideas. They represent intentional user scope that never becomes a task because it was never promoted to `pages/*.html` and was never added to `phases_inventory`.

**Fix:** After reading `## Background extracted ideas`, for each idea entry:
- Check if the idea maps to an existing task in phases_inventory or the UI Pages → Component Map
- If NOT covered: surface a gate question before Step 7: "Background UI idea '{title}' has no implementation task. Options: (a) promote to UI Direction now (generate page), (b) add as implementation task with 'design-TBD' note, (c) mark as descoped."
- Record the resolution in working notes before ROADMAP generation

### Gap 8 — No architect decision → UI prototype freshness check
**Source:** `notes.md ## decisions[]` records architectural decisions. When arch_to_ui_sync entries have `status: noted` it means the UI prototype was created BEFORE those decisions and may not reflect them.

**Current:** crystallize reads decisions[] and writes them to ARCHITECTURE.md. It reads pages/*.html and extracts components. But there is **no check** that pages/*.html actually reflects the architectural decisions in decisions[].

**Example:** Architect decided "real-time WebSocket for notifications" after landing.html was created. landing.html shows a simple "bell icon" with no real-time indicator. crystallize imports both artifacts but never flags the mismatch.

**Fix:** After reading both decisions[] and pages/*.html, cross-check: for any decision with a UI implication (from arch_to_ui_sync table triggers) — verify the relevant page/*.html has a visible representation of that decision. If not: surface a "design staleness" warning and create a task `update {page}.html to reflect {decision}` before the component implementation task.

## Discussion
This bug pattern will repeat for every project that uses `--ui` mode: brainstorm creates rich prototypes, crystallize reads them but doesn't bind them to tasks, vp-auto creates minimal stubs that satisfy routing/wiring requirements without implementing the design. The fix closes the loop: read → bind → enforce → implement → gate.

**Secondary symptom:** Even when pages ARE in phases_inventory, vp-auto may implement a "minimum viable" version without referencing the prototype because the task file doesn't point to it. The `## UI Prototype Reference` field addresses this secondary case.
