## Conflict Analysis

### ⚠️ System Diagram vs architecture.html
`architecture.html` hiện đã có `graph TD` system diagram + component table.

**Đánh giá:**
- `architecture.html` focus: **logical architecture** — component boundaries, services, layers.
- "System diagram" của user có thể muốn: **C4 Context diagram** — system trong bối cảnh người dùng và external systems (khác với internal components).

**Quyết định đề xuất:** **Enhance `architecture.html`** thay vì tạo trang mới trùng lặp:
- Thêm tab/section `C4Context` (system-level) bên cạnh `graph TD` (component-level)
- Thêm table: **External Systems** (System | Type | Description | Interface)
- Không tạo `system-diagram.html` riêng → tránh duplicate

### ⚠️ Sequence Diagram vs data-flow.html
`data-flow.html` hiện có `sequenceDiagram` placeholder + `flowchart LR` (event flow).

**Đánh giá:**
- `data-flow.html` = **high-level flows**: service-to-service, async event queues, message routing — 1–2 diagrams tổng quan.
- `sequence-diagram.html` = **scenario-level detail**: từng use case / API call step-by-step, nhiều diagrams cho nhiều scenario khác nhau.

**Quyết định đề xuất:** **Tạo `sequence-diagram.html` mới** (không xóa data-flow.html):
- `data-flow.html`: giữ nguyên — high-level service flows + async events
- `sequence-diagram.html`: mới — per-scenario sequences (Login flow, Payment flow, v.v.), nhiều `sequenceDiagram` blocks có header
- Ghi rõ differentiation trong workflow docs để AI không nhầm lẫn
