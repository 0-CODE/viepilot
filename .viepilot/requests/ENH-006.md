## Summary
`vp-audit` hiện tại audit **bản thân framework viepilot** (kiểm tra skills/, workflows/, ARCHITECTURE.md, docs/skills-reference.md). Khi user dùng viepilot để build dự án khác, `vp-audit` phải audit **dự án đó**, không phải framework.

Cần tách thành 2 chế độ rõ ràng:
1. **Project audit** (mặc định) — audit dự án người dùng đang build
2. **Framework audit** (internal) — audit bản thân viepilot framework

## Problem Analysis

### Hiện tại `vp-audit` đang làm gì?

| Check | Phục vụ | Có ý nghĩa cho user project? |
|-------|---------|------------------------------|
| `ls skills/vp-*/` count vs ARCHITECTURE.md | viepilot framework | ❌ User project không có skills/ |
| `ls workflows/` count | viepilot framework | ❌ User project không có workflows/ |
| `node bin/vp-tools.cjs help` | viepilot CLI | ❌ User project không có vp-tools |
| README.md `skills-N` badge drift | viepilot framework | ❌ User README không có badge này |
| `docs/skills-reference.md` sections | viepilot framework | ❌ Đây là file của viepilot |
| `your-org` placeholder URLs | viepilot docs | ⚠️ Có thể relevant nếu user project có docs |
| `.viepilot/ROADMAP.md` phase status | viepilot tracking | ✅ Relevant — đây là state của project |

### Điều `vp-audit` NÊN làm khi user audit project của họ

`vp-audit` trong context **autonomous/vibe coding** nên kiểm tra:

#### 1. ViePilot state consistency (luôn applicable)
- `.viepilot/TRACKER.md` current state vs `.viepilot/phases/*/PHASE-STATE.md`
- `.viepilot/ROADMAP.md` phase status vs PHASE-STATE.md
- `.viepilot/HANDOFF.json` vs TRACKER.md (resume-state consistency)
- Git tags `vp-p{N}-complete` vs PHASE-STATE.md complete phases

#### 2. Project documentation drift (applicable to any project)
- `README.md` có up-to-date với thực tế không?
  - Version đề cập có khớp với `package.json`/`pom.xml`/`pyproject.toml`?
  - Installation instructions có còn đúng không?
  - Features list có phản ánh đúng code không?
- Docs folder có missing documentation cho new features (được thêm trong phases gần đây)?
- `CHANGELOG.md` có reflect đúng commits gần đây không?

#### 3. Code quality drift (optional, project-type aware)
- TODO/FIXME comments còn trong code không?
- Dead code / unused imports (nếu project có linter)?
- API docs outdated (nếu project là API)?

#### 4. Framework integrity (chỉ chạy khi `--framework` hoặc trong viepilot repo)
- Detect: `ls skills/vp-*/SKILL.md` → nếu tồn tại → viepilot framework repo
- Nếu là framework: chạy checks hiện tại (ARCHITECTURE.md, skills-reference.md, etc.)
- Nếu không: bỏ qua hoàn toàn

### Thay đổi trong `workflows/audit.md`

#### Step 0: Detect project type (thêm mới)
```bash
IS_VIEPILOT_FRAMEWORK=false
if [ -d "skills" ] && ls skills/vp-*/SKILL.md 2>/dev/null | head -1 > /dev/null; then
  IS_VIEPILOT_FRAMEWORK=true
  echo "→ Detected: ViePilot framework repository"
else
  echo "→ Detected: User project using ViePilot"
fi
```

#### Step 1: ViePilot State Consistency (rewrite)
Replace current "Collect Actual Implementation State" với state consistency checks:
```bash
# Check TRACKER.md vs PHASE-STATE.md
TRACKER_PHASE=$(grep "Current Phase" .viepilot/TRACKER.md | ...)
PHASE_STATE=$(grep "Status" .viepilot/phases/.../PHASE-STATE.md | ...)
# Compare and report mismatches
```

#### Step 2: Project Documentation Drift (rewrite)
Replace "Parse ARCHITECTURE.md" với generic docs check:
```bash
# Detect version file
if [ -f "package.json" ]; then
  PROJECT_VERSION=$(node -p "require('./package.json').version")
elif [ -f "pom.xml" ]; then
  PROJECT_VERSION=$(grep -m1 "<version>" pom.xml | sed 's/.*<version>//;s/<.*//')
elif [ -f "pyproject.toml" ]; then
  PROJECT_VERSION=$(grep '^version' pyproject.toml | cut -d'"' -f2)
fi
# Compare README.md version mentions vs actual
```

#### Step 3: Framework Integrity (conditional)
```bash
if [ "$IS_VIEPILOT_FRAMEWORK" = true ]; then
  # Run existing ARCHITECTURE.md checks
  # Run docs/skills-reference.md checks
  # Run README.md viepilot-specific badge checks
fi
```
