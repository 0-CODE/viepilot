# ENH: vp-brainstorm — Idea-to-Architecture breakdown loop (Architect + UI Direction parity)

## Meta

- **Status**: triaged
- **Priority**: high
- **Created**: 2026-04-18
- **Reporter**: User
- **Assignee**: AI

## Summary
The current `vp-brainstorm` workflow collects ideas well but lacks a structured **breakdown loop** that explicitly maps each idea/feature in `## Phases` to both:
1. **Architect components** (which pages in the architect workspace handle this?)
2. **UI Direction screens** (which screens/flows implement this?)

The two modes (Architect Design and UI Direction) run largely independently. There is a one-way bridge (UI → Architect via ENH-034 `architect_delta_sync`) but no Architect → UI bridge, no cross-coverage validation, and no explicit "are we done breaking this down?" checkpoint before handing off to `/vp-crystallize`.

## Details

### Current behavior gaps

**Gap 1 — No idea→component→screen mapping step**
After a topic is scoped and `## Phases` is populated, the workflow moves directly toward saving. There is no explicit step that says:
> "For each feature in Phase 1, which architect component owns it? Which UI screen surfaces it?"

**Gap 2 — One-way Architect ↔ UI bridge**
`architect_delta_sync` (ENH-034) syncs UI session → Architect pages.
But the reverse is missing: when an architectural decision is made (e.g., "async event queue for notifications"), the UI Direction is not automatically prompted to add the relevant feedback states, loading screens, or error flows.

**Gap 3 — No pre-crystallize completeness check**
Before `/end` or `/save`, there is no validation that confirms:
- All Phase 1 features have a named architect component
- All Phase 1 features have a corresponding UI screen/flow
- Architect workspace and UI Direction workspace are mutually consistent

**Gap 4 — No loop for iterative refinement**
If during architect design a new constraint emerges (e.g., "we can't support real-time sync — must be polling"), there is no loop that brings the user back to re-evaluate the UI Direction decisions that assumed real-time behavior.

**Gap 5 — Breakdown ordering is implicit**
Brainstorm collects ideas freely. When to start Architect mode vs. UI Direction mode is left entirely to the user or to keyword heuristics. There is no recommended ordering or explicit "breakdown phase" in the workflow.

### Expected behavior (proposed improvements)

#### A. Feature → Coverage mapping step (new step after scope locked)
After `## Phases` is confirmed, run a structured coverage check:
```
For each feature in Phase 1:
  → Architect: which component/page covers this? [architecture / data-flow / erd / apis / ...]
  → UI: which screen/flow surfaces this? [page slug or "not yet modeled"]
Output: coverage matrix in ## Coverage section of notes.md
```

#### B. Architect → UI Direction feedback loop
When the Architect workspace is updated (any page edit), trigger a reverse check:
> "This architectural decision may affect these UI screens: [...]. Do you want to update UI Direction to reflect it?"
(Symmetric to architect_delta_sync — call it `ui_direction_delta_sync` or extend ENH-034)

#### C. Pre-crystallize breakdown completeness gate
Before `/end` or `/save as Completed`, validate:
- [ ] Every Phase 1 feature is in the coverage matrix
- [ ] No feature is listed as "not yet modeled" in both columns (must have at least Architect OR UI coverage)
- [ ] Warn if Architect and UI Direction are inconsistent (e.g., API in architect not surfaced in any UI screen)

#### D. Recommended breakdown ordering guidance
Add an explicit **recommended flow** for new sessions:
```
1. Collect ideas freely (current behavior)
2. Lock scope → assign to Phases (current behavior)
3. [NEW] Feature → Coverage mapping: architect component + UI screen per feature
4. [NEW] Architect Design phase: fill architect workspace per coverage matrix
5. [NEW] Architect → UI sync: check which UI screens need updating based on arch decisions
6. UI Direction phase: fill UI workspace per coverage matrix + arch feedback
7. [NEW] Completeness gate: validate all Phase 1 features have coverage
8. /save → /vp-crystallize (current behavior)
```
