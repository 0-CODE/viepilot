## Summary
Brainstorm hiện tại khai thác sâu về domain/architecture/data/API/infra nhưng **hoàn toàn bỏ qua Admin & Governance** — bao gồm admin panel, user management, monitoring dashboard, audit logs, access control, billing, và observability. Khi brainstorm không elicit các yếu tố này, vp-crystallize cũng không sinh ra admin-related phases/tasks → project thiếu back-office capabilities từ đầu.

## Details

### Gap analysis

**Current brainstorm Topics Template (5 topics):**
1. Domain Analysis
2. Architecture
3. Data Model
4. API Design
5. Infrastructure

**Not covered:**
- Admin panel / back-office interface (quản trị nội dung, user management UI)
- User & Role management (CRUD users, RBAC/ABAC, permission assignment)
- Monitoring & Observability (system health, metrics, alerting, tracing)
- Audit logs & Activity tracking (ai làm gì, khi nào, thay đổi gì)
- Billing & Subscription management (plans, invoices, payment history)
- Rate limiting & Quota management (per-user/tier limits)
- Feature flags & System configuration (runtime toggles, env settings)
- Reporting & Analytics (business metrics, usage reports, export)
- Notification management (email/push templates, delivery settings)

**Impact on crystallize:**
- `PROJECT-CONTEXT.md` không có `## Admin & Governance` section
- `ARCHITECTURE.md` không có admin services/components
- Phase planning bỏ qua admin phases → được discover muộn khi đã code (tốn kém)

### Desired improvement

#### A — New "Admin & Governance" topic in brainstorm Topics Template (Topic 6)
```
6. **Admin & Governance**
   - Who are the admin personas? (super-admin, org-admin, support agent)
   - User & Role management needed? (invite, deactivate, RBAC)
   - Monitoring dashboard? (uptime, error rates, latency, active users)
   - Audit log requirements? (what actions must be tracked? retention?)
   - Billing/subscription management? (plans, invoices, downgrade/upgrade)
   - Rate limiting / quota per tier? (API limits, storage limits)
   - Feature flags / runtime config? (kill switches, A/B toggles)
   - Reporting & data export? (CSV export, business metrics dashboard)
   - Notification management? (template editor, delivery log, unsubscribe)
```

#### B — Proactive elicitation heuristic (keyword triggers)
When user mentions any of these keywords during brainstorm:
> `user`, `role`, `permission`, `admin`, `back-office`, `dashboard`, `monitor`, `audit`, `log`, `billing`, `subscription`, `plan`, `rate limit`, `quota`, `feature flag`, `config`, `report`, `analytics`, `export`, `notification`, `alert`, `health`, `SLA`, `compliance`

→ Proactively prompt:
```
🔐 I noticed your project may need admin/governance capabilities.
Should we discuss:
1. Yes — explore Admin & Governance topic (recommended for multi-user products)
2. Not needed — single-user or admin-free project
3. Later — add to notes, continue current topic
```

#### C — Admin coverage gate before /save
After Phase assignment is complete, check if project has:
- Multi-user indicators (roles, teams, organizations)
- OR SaaS/paid product indicators (subscription, billing, plan)
- OR compliance/enterprise indicators (audit, GDPR, SOC2)

If any detected AND no admin topic discussed → warn (non-blocking):
```
⚠️ Admin gap detected: your project has multi-user/SaaS signals but no Admin & Governance topic was covered.
Consider: /add-topic admin  OR  ignore (admin can be added in a later phase)
```

#### D — Architect workspace: `admin.html` page
When Admin & Governance topic is discussed AND Architect Mode is active:
- Add `admin.html` page to architect workspace
- Content: admin actor flow, role hierarchy, key admin operations list, audit schema
- Trigger keywords: `admin panel`, `back-office`, `RBAC`, `user management`

#### E — Crystallize integration
`PROJECT-CONTEXT.md` gets new section `## Admin & Governance` (from notes.md `## admin` YAML):
```markdown
## Admin & Governance

| Capability | Required | Phase | Notes |
|-----------|----------|-------|-------|
| User management UI | yes | 2 | invite, deactivate, role assign |
| Audit log | yes | 3 | all write operations, 90-day retention |
| Monitoring dashboard | optional | 3 | error rate, latency, active users |
| Billing management | no | — | not in scope |
```

`ARCHITECTURE.md` gets admin service components if applicable.

### Why this matters
- Admin capabilities are **discovered late** in most projects → expensive rework
- Multi-user SaaS products need admin from day 1 (support tooling, compliance)
- Monitoring/observability must be planned in architecture phase, not retrofitted
- Brainstorm is the right time to ask — before crystallize locks the phase plan
