# BUG: vp-auto can execute before enforcing full task documentation flow

## Meta

- **Reporter**: User

## Summary
`/vp-auto` in current behavior can proceed with implementation first, then backfill task/documentation/state updates, which increases documentation drift risk.

## Details

### Actual behavior
- In some runs, execution starts before the task artifact/state update path is fully enforced as a hard gate.
- Documentation/state files may be updated after code execution, causing temporary or persistent drift.

### Expected behavior
- For each task/sub-task, enforce strict order:
  1. Validate task contract and required documentation sections.
  2. Update planning/state artifacts required before execution.
  3. Execute implementation.
  4. Verify quality gates.
  5. Update completion artifacts immediately (state-first).

### Impact
- Higher chance of drift between implementation and docs/state (`PHASE-STATE`, `TRACKER`, `HANDOFF`, task notes).
- Harder resume/debug trace if interruptions happen between implementation and backfilled docs.

## Acceptance Criteria
- [x] `vp-auto` blocks implementation when task documentation contract is incomplete.
- [x] Workflow wording clearly enforces pre-execution documentation/update gates.
- [x] Task execution order is explicit and testable in docs: plan/docs first, then code.
- [x] Add/extend checks in audit to flag "execute-first, docs-later" pattern.

Issue is about enforcement consistency, not intent. Existing docs mention state-first and task contract, but behavior should hard-stop earlier and more predictably when preconditions are not met.
