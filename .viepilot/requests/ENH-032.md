## Summary
Add a language configuration system to ViePilot: installer prompts for communication language and document language during setup; crystallize/brainstorm/autonomous workflows read these settings at runtime to produce output in the configured languages.

## Details

### 1. Installer Setup (New)

During `npm install -g viepilot` / `vp-tools install`, prompt the user:

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 VIEPILOT ► LANGUAGE SETUP
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

 Communication language (AI responses, prompts, messages):
   [1] English  (default)
   [2] Vietnamese
   [3] Other (enter BCP-47 code, e.g. ja, zh-TW, fr)

 Document language (generated docs: README, CHANGELOG, ROADMAP, etc.):
   [1] English  (default)
   [2] Vietnamese
   [3] Other (enter BCP-47 code)
   [4] Same as communication language

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

Defaults (if user presses Enter): both = `en`.

### 2. Config Storage

Persist to `~/.claude/viepilot/config.json` (global, not per-project):

```json
{
  "language": {
    "communication": "en",
    "document": "en"
  }
}
```

Also expose a `vp-tools config set language.communication vi` CLI command for post-install reconfiguration.

### 3. Runtime Behavior — Crystallize

`/vp-crystallize` reads `config.language` and:
- Uses `language.communication` for all AI messages, prompts, status output during the session
- Uses `language.document` when generating/writing:
  - `README.md`, `CHANGELOG.md`, `ROADMAP.md`
  - `ARCHITECTURE.md`, `AI-GUIDE.md`, `CONTRIBUTING.md`
  - Phase files: `SPEC.md`, `TASK.md`, `VERIFICATION.md`, `PHASE-STATE.md`
  - Any other `.viepilot/*.md` tracking files

Inject into crystallize workflow context:

```
COMMUNICATION_LANG = config.language.communication   # e.g. "vi"
DOCUMENT_LANG      = config.language.document        # e.g. "en"

→ All AI responses/prompts in COMMUNICATION_LANG
→ All written .md files in DOCUMENT_LANG
```

### 4. Runtime Behavior — Brainstorm

`/vp-brainstorm` uses auto-detection:
- Detect the language the user is actively writing in during the session
- Save brainstorm output files (`.viepilot/brainstorm/*.md`) in that detected language
- Do **not** force `config.language.document` for brainstorm files — brainstorm content is exploratory and should match the conversation

Rule:
```
brainstorm_file_language = detect(user_conversation_language)
# Fallback: config.language.communication if detection is ambiguous
```

### 5. Runtime Behavior — Autonomous / vp-auto

`/vp-auto` uses `config.language.communication` for:
- Phase start/end banners
- Control point prompts
- Error/blocker messages
- Task status updates

### 6. Config CLI (vp-tools)

```bash
vp-tools config get                         # Show current config
vp-tools config set language.communication vi
vp-tools config set language.document en
vp-tools config set language.communication en  # Reset to default
```

## Discussion

### Why separate communication vs document language?
A Vietnamese developer may want to chat with ViePilot in Vietnamese (communication = vi) but produce English documentation for an international team (document = en). These are independent concerns.

### Why brainstorm uses conversation detection instead of config?
Brainstorm sessions are exploratory scratch space. When a user writes in Vietnamese during brainstorm, storing the output in English would require real-time translation and risk losing nuance. Auto-detecting preserves fidelity.

### Config location: global vs per-project?
Global (`~/.claude/viepilot/config.json`) — language preference is a user preference, not a project setting. A developer switches language preference once, not per-repo. Projects can override via `.viepilot/config.json` (local override, lower precedence than explicit user action, higher than global default) if needed in the future.
