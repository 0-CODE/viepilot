## Summary
When a URL is provided to `vp-request` (GitHub Issue, Jira ticket, Linear issue, Notion page, Trello card), automatically browse that URL using `browser-intake-agent` and pre-fill the request form (title, description, labels, priority, type) — so the user only needs to confirm rather than re-type from the external source.

## Problem Statement
When a user wants to log an issue from an existing tracking system (GitHub Issues, Jira, Linear, Notion, Trello), they currently must:
1. Open the external tool
2. Read the ticket
3. Manually retype title, description, steps, severity into `/vp-request` prompt

This is duplicated effort that discourages cross-system intake.

## Proposed Solution

### URL detection in vp-request Step 1
If `{{VP_ARGS}}` contains a URL pattern → trigger "URL enrichment mode" before Step 2:

```
URL detected: https://github.com/org/repo/issues/123
Fetching context...
```

### Dispatch browser-intake-agent for known URL patterns
URL routing:
- `github.com/*/issues/*` → GitHub Issues parser (title, body, labels, assignee)
- `linear.app/*/issue/*` → Linear issue parser
- `jira.atlassian.net/browse/*` → Jira public ticket parser
- `trello.com/c/*` → Trello card parser
- `notion.so/*` → Notion page parser
- `*.atlassian.net/servicedesk/customer/portal/*` → Jira Service Desk
- Generic fallback → open URL, snapshot, extract structured content

### Pre-fill behavior
After extraction:
```
Context extracted from: https://github.com/org/repo/issues/123

  Title:       Login page crashes on Safari iOS 17
  Type:        Bug (detected from: "bug" label)
  Description: Steps to reproduce: 1. Open Safari...
  Priority:    high (detected from: "priority:high" label)

Confirm or edit? [y/edit]
```

On confirm → skip redundant questions in Step 2, jump directly to acceptance criteria + routing.

### Fallback
If `browser-intake-agent` not available or URL doesn't match known patterns → proceed with normal vp-request flow, show URL as a "Related link" field in the request file.
