## Summary
`vp-brainstorm` is designed around web/app project archetypes. When applied to **Embedded / Firmware projects** (MCU-based, IoT, bare-metal or RTOS), 10 structural gaps cause the session to miss critical domain knowledge, generate irrelevant artifacts (UI Direction on an LCD keyword), and produce phase plans that don't match firmware development flow.

## Current Behavior

The current brainstorm feature set assumes a web/backend/mobile product:
- Architect workspace covers software components only (C4, ERD, APIs, Deployment)
- Tech-stack probes ask about frameworks, databases, cloud infra — no MCU/toolchain questions
- UI Direction background extraction fires on `display`, `screen`, `LCD`, `OLED` keywords — all common in embedded but referring to hardware, not web UI
- Phase assignment is generic (Phase 1, 2, 3...) with no firmware-layer ordering awareness
- No topics for RTOS scheduling, pin/GPIO mapping, memory partitioning, power budget, bus protocol matrix, or safety compliance

## Desired Behavior

When the user discusses an Embedded/Firmware project (detected by MCU/RTOS/firmware/GPIO/bare-metal keywords, or by explicit `--domain embedded` flag), `vp-brainstorm` activates an **Embedded Domain Mode** that:

1. Adds domain-specific workspace pages
2. Suppresses irrelevant web-UI triggers
3. Probes embedded-specific topics (MCU, toolchain, RTOS, safety)
4. Orders phase assignment by firmware development layers

### Gap 9 — Firmware Phase Ordering
**Missing**: Phase assignment uses generic Phase 1/2/3 ordering with no firmware-layer awareness.
**Need**: When embedded domain is active, brainstorm suggests standard firmware phase template:
```
Phase 1: Board Bring-Up (clock, GPIO, UART console, LED blink)
Phase 2: Driver Layer (peripheral drivers for all hw-topology items)
Phase 3: RTOS Configuration (task creation, queues, semaphores)
Phase 4: Middleware & Protocols (MQTT/BLE/LoRa stack, filesystem, OTA bootloader)
Phase 5: Application Logic (state machines, business logic, user interface)
Phase 6: Integration & Testing (hardware-in-the-loop, timing verification)
Phase 7: OTA & Production (bootloader signing, provisioning, factory test)
```
User can customize/remove phases, but the template ensures coverage of typical embedded concerns.

### Gap 10 — Safety/Compliance Topic
**Missing**: Safety requirements are never elicited for safety-critical embedded.
**Need**: Optional safety topic (triggered by `safety`, `watchdog`, `fault`, `SIL`, `ASIL`, `IEC`, `ISO 26262`, `DO-178` keywords or explicit user mention):
- Safety level: IEC 61508 SIL 1–4 / ISO 26262 ASIL A–D / DO-178C / none
- Watchdog configuration (IWDG/WWDG, timeout, pet strategy)
- Stack overflow detection (MPU, canary, FreeRTOS stack check)
- Fault handler strategy (HardFault, MemManage, BusFault)
- Safe state definition (what happens on error)
**notes.md section**: `## safety_config` YAML
**crystallize export**: `## Safety & Reliability` in ARCHITECTURE.md

- Suppress web UI triggers (Gap 6)
- Inject embedded topic probes (Gaps 2, 3, 7, 10)
- Add hardware pages to Architect workspace (Gaps 1, 4, 5, 8)

## Discussion
The current brainstorm is strongly web/app-centric. Embedded projects have fundamentally different architecture artifacts (hardware topology, pin maps, memory layout) and development phases (bring-up → drivers → RTOS → application). Without these, a brainstorm session for an STM32 IoT sensor node would: (1) suggest UI Direction on "OLED display" → irrelevant, (2) miss MCU/toolchain decisions → crystallize produces a software-only ARCHITECTURE.md with no hardware sections, (3) generate phase plans that don't cover Board Bring-Up or OTA bootloader design. The 10 gaps can be addressed as a single "Embedded Domain Mode" feature toggle, keeping the existing web-app behavior intact.
