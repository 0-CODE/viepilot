# ENH: Unify version bump orchestration across autonomous.md and evolve.md

## Summary
Version bump rules are defined independently in `autonomous.md` and `evolve.md`, creating
risk of contradictions when a phase contains mixed change types. The rules should resolve
from a single authoritative source.

## Details

### Current split
- `autonomous.md`: "Features added → MINOR, Fixes only → PATCH, Breaking → MAJOR"
- `evolve.md` Add Feature mode: "Suggest MINOR version bump"
- `evolve.md` refactor table: "Refactor → PATCH (no behavior change)"
- `evolve.md` bug mode: "Patch bump"

### Conflict scenario
1. `evolve.md` creates Phase 82 for a critical bug → suggests PATCH → sets `next_version: 2.17.1`
2. User runs the phase but also adds a small convenience feature along the way
3. `autonomous.md` bump rule sees new feature → escalates to MINOR → sets `2.18.0`
4. Both workflows believe they own the version decision; the final version depends
   on execution order and which rule the AI applies last

### Desired state
- Single version bump rule table in ONE location (e.g., `SYSTEM-RULES.md` or a new
  `docs/developer/versioning.md`)
- `autonomous.md` and `evolve.md` reference that source, do NOT redefine rules inline
- Clear precedence: MAJOR > MINOR > PATCH; mixed phase → highest applies

## Acceptance Criteria
- [ ] Version bump rules consolidated into one canonical location
- [ ] Both `autonomous.md` and `evolve.md` reference that location instead of inline rules
- [ ] "Mixed phase" scenario documented: if feature + fix → MINOR wins
- [ ] No contradictory bump guidance between the two workflows
- [ ] Contract test asserting bump escalation logic
