# ENH: vp-qa — Project-Level QA Agent Team Generator

## Summary

Add a new `vp-qa` skill to ViePilot that generates a complete, context-aware backend QA agent
team into any target project's `.claude/agents/` directory. The generated agents (orchestrator
+ 4 specialist subagents) scan the backend codebase for completeness issues, security
vulnerabilities, performance bottlenecks, and context drift — then automatically create
`vp-request` entries for found issues, with interactive accept/decline via `AskUserQuestion`.

## Origin

Inspired by user's existing manual prompt:

> `/vp-request` thực hiện tạo agent và subagent vào `.claude` ở cấp project và đáp ứng kiến
> trúc của claude. Mục đích tôi muốn lập 1 team QA agents toàn diện cho backend để có thể
> scan tất cả vấn đề source code backend từ stub/placeholder, incomplete đến các vấn đề bảo
> mật của source code, các điểm nghẽn khi ở tải cao, project context audit với codebase và
> các vấn đề khác.

The manual prompt works but requires the user to manually specify agent structure, stack context,
and vp-request integration each time. `vp-qa` makes it a first-class ViePilot skill: one command,
fully tailored to the project's actual stack.

## Feature Design

### Design Principle: Scan-First, Generate-Second

**Key correction from initial design**: The skill must NOT generate agents from hardcoded
templates. Instead, `vp-qa` must first **research the target project** — read codebase
structure, understand the actual stack, technology choices, directory layout, and existing
issues — and then **dynamically decide**:
- How many specialist subagents are appropriate
- What each subagent's scope should cover
- Which specific patterns to scan for, given the detected stack
- How to prioritize given project size and complexity

This is analogous to how a senior QA engineer would assess a new codebase before designing a
test strategy — not apply a generic checklist, but understand the system first.

### Invocation
```
/vp-qa                → research codebase → generate context-tailored QA team
/vp-qa --run          → generate + immediately run qa-orchestrator scan
/vp-qa --focus sec    → research + bias toward security agents
/vp-qa --focus perf   → research + bias toward performance agents
```

### Phase 1: Research (before any agent generation)

When invoked, the skill first performs a **codebase research scan**:

```
1. Read .viepilot/PROJECT-META.md, STACKS.md, PROJECT-CONTEXT.md (if present)
2. Detect stack from package.json / pom.xml / go.mod / requirements.txt / Gemfile
3. Read directory tree: find src/, app/, lib/, services/, api/, routes/, etc.
4. Sample 5-10 representative source files to understand coding patterns
5. Check for existing test files: coverage hints
6. Check for existing .viepilot/requests/ to avoid duplicating known issues
7. Note project size (file count, service count, DB layer presence)
```

Based on research output, `vp-qa` decides:
- Which scan domains are relevant (a static site doesn't need N+1 query scanning)
- How many subagents to generate (2 for a small project, 5+ for a large monolith)
- What patterns matter for this specific stack version (Express 4 vs Fastify vs NestJS)
- Which directories to target vs exclude

### Phase 2: Generate agents

Based on Phase 1 findings, generate **contextual agent files** in `.claude/agents/`:
- Always: `qa-orchestrator.md` (the driver)
- Conditionally: specialist subagents based on what was found

Example specialist subagents (actual list determined by research):
- `qa-completeness-scanner` — stubs, TODO, placeholders, incomplete handlers
- `qa-security-scanner` — OWASP Top 10, hardcoded secrets, missing auth, weak crypto
- `qa-performance-scanner` — N+1, blocking I/O, missing indexes, unbounded loops
- `qa-context-auditor` — dead code, API drift, missing tests, circular deps
- Additional domain-specific agent if needed (e.g. `qa-migration-auditor` if many DB migrations found)

Each agent's system prompt includes:
- The specific file paths / directories to target
- Stack-specific anti-patterns to look for
- What "done" means for that agent (structured issue report format)

### Generated Agent: `qa-orchestrator`

The orchestrator is the main agent the user invokes. Its behavior when run in the target project:

```
1. Fan out to all 4 subagents in parallel
2. Collect structured issue lists from each
3. Group issues by severity (critical / high / medium / low)
4. For critical/high: AskUserQuestion — accept (→ create vp-request) or decline (skip)
5. For medium/low: batch accept/decline or auto-create
6. Create vp-request entries for accepted issues
7. AskUserQuestion: "N issues logged. What next?"
   → Run /vp-evolve to plan fixes  |  Done for now
```

### Generated Agent: `qa-completeness-scanner`

Scans for:
- `TODO`, `FIXME`, `HACK`, `XXX`, `PLACEHOLDER` comments
- `NotImplementedException`, `throw new Error('not implemented')`
- Empty function bodies: `function foo() {}`, `async handler() { return null }`
- Stub patterns: `return null; // TODO`, `return [];`
- Incomplete migrations: schema files with `-- TODO` blocks
- Dead route handlers (routes defined but handler is stub)

### Generated Agent: `qa-security-scanner`

Scans for (OWASP Top 10 + common backend patterns):
- SQL injection: string-concatenated queries, raw SQL without parameterization
- Hardcoded secrets: API keys, passwords, connection strings in source
- Missing authentication middleware on sensitive routes
- Broken access control: no ownership check before update/delete
- Insecure deserialization: `JSON.parse(userInput)` without validation
- Missing rate limiting on auth endpoints
- Weak cryptography: MD5/SHA1 for passwords, `Math.random()` for tokens
- Missing input validation / sanitization
- Exposed stack traces in error responses
- Missing CORS restrictions

### Generated Agent: `qa-performance-scanner`

Scans for:
- N+1 query patterns: DB call inside a loop
- Missing pagination on list endpoints
- Synchronous blocking in async context (`fs.readFileSync` in request handler)
- Missing DB indexes on frequently-queried foreign keys
- No query timeout / no connection pool limits
- In-memory accumulation on large datasets
- Missing caching on expensive repeated computations
- Unbounded recursion or while(true) without circuit breaker
- Heavy computation on main thread (no worker offload)

### Generated Agent: `qa-context-auditor`

Scans for:
- Dead code: exported functions with 0 references
- API contract drift: documented endpoints that don't exist in routes, or vice versa
- Missing test files: src files with no corresponding `.test.` file
- Environment variable usage without validation or .env.example entry
- Deprecated dependency versions (check package.json against known EOL)
- Circular dependencies
- Missing error boundaries / unhandled promise rejections
- Inconsistent naming conventions (camelCase vs snake_case mixing)
- Architecture violations: e.g., controller directly calling DB (no service layer)

### Stack-Aware Tailoring

Each subagent's prompt is specialized by stack:

| Stack | Completeness extras | Security extras | Performance extras |
|-------|---------------------|-----------------|-------------------|
| Node.js | Express route stubs, missing middleware | `eval()`, prototype pollution | Event loop blocking, missing `await` |
| Java/Spring | `@NotImplemented`, empty `@Override` | OGNL injection, XXE | N+1 with Hibernate, missing `@Async` |
| Python/Django | `pass` in view handlers, `raise NotImplementedError` | SSTI, pickle deserialization | Sync ORM in async view, missing `select_related` |
| Go | empty interface implementations | goroutine leaks, missing context cancellation | Unbuffered channel deadlock, missing connection reuse |

### vp-request Integration

When the orchestrator finds issues, it creates properly formatted vp-request entries:

```markdown
# BUG: [Completeness] 12 stub handlers in src/api/payments/

## Meta
- ID: BUG-{N}
- Type: Bug
- Severity: high
- Source: qa-completeness-scanner
- Files: src/api/payments/handler.js:45, :112, ...

## Summary
12 route handlers return `return null; // TODO` in the payments module.
These will 500 in production if reached.
...
```

## Technical Notes

### Agent frontmatter (target project agents — NOT global install)
```yaml
---
name: qa-orchestrator
description: Backend QA orchestrator. Fans out to completeness, security, performance,
  and context subagents; collects issues; creates vp-request entries; interactive
  accept/decline via AskUserQuestion.
model: claude-sonnet-4-6
maxTurns: 50
permissionMode: auto
tools:
  - Read
  - Bash
  - Glob
  - Grep
  - LS
  - Agent
  - AskUserQuestion
---
```

### Subagent frontmatter (read-only scanners)
```yaml
---
model: claude-haiku-4-5
maxTurns: 20
permissionMode: auto
tools: [Read, Bash, Glob, Grep, LS]
disallowedTools: [Edit, Write, WebSearch, WebFetch]
---
```

### Generation flow in vp-qa skill
```
vp-qa invoked
  → read STACKS.md / auto-detect stack
  → read project dir structure (find src, app, lib, etc.)
  → render 5 agent templates with stack context substituted
  → write to .claude/agents/{name}.md
  → AskUserQuestion: run now / done
```

### Adapter-Aware Output (Multi-Adapter Support)

`vp-qa` must generate output that matches the **active adapter's agent format**.
Claude Code's `.claude/agents/` is only one of 5 possible targets:

| Adapter | Output format | Multi-agent support |
|---------|--------------|---------------------|
| **Claude Code** | `.claude/agents/qa-orchestrator.md` + subagent `.md` files | ✅ Full fan-out via `Agent` tool (parallel subagents) |
| **Cursor** | `.cursor/rules/qa-checklist.mdc` | ❌ Single-rule, sequential inline scanning |
| **Codex** | Appends `## QA Agent Instructions` section to `AGENTS.md` | ❌ Single-agent, sequential scan (no Agent tool) |
| **Antigravity** | `.agents/skills/qa-orchestrator/SKILL.md` + sub-skills | ⚠️ Single-level only (no nested Agent) |
| **GitHub Copilot** | `.github/agents/qa-orchestrator.agent.md` | ⚠️ Main agent only (no subagent dispatch) |

**Execution model difference:**
- Claude Code: true parallel fan-out (orchestrator spawns 4+ subagents simultaneously)
- All other adapters: sequential single-agent scanning (all domains covered in one pass)

The **scan domains** (completeness, security, performance, context) are identical across
adapters — only the execution model differs. Codex/Cursor/Copilot agents run through all
domains sequentially in one prompt.

**Detection**: `vp-qa` calls `adapter-context.cjs` (already in `lib/`) to detect the active
adapter and route output accordingly. `--target <adapter>` flag overrides detection.

## Discussion

Key design decisions:
1. **Template engine**: Use plain string replacement (`{{STACK_RULES}}` blocks) in agent `.md`
   files — no external template dependency.
2. **Target project vs ViePilot repo**: `vp-qa` writes to the *calling project's* `.claude/agents/`,
   not to `~/.claude/agents/` (global). This keeps QA agents project-scoped.
3. **vp-request format**: The orchestrator creates files in `.viepilot/requests/` of the target
   project — assumes ViePilot is initialized there (`/vp-init` required first).
4. **Severity routing**: Critical = blocking AskUserQuestion; High = confirmable; Medium/Low =
   auto-create with one batch confirm prompt.
5. **Re-run safety**: `vp-qa` overwrites existing agent files with fresh stack-aware content on
   re-run (not append). Existing `.viepilot/requests/` files are never overwritten.
