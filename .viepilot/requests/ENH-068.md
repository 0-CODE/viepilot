## Summary

The current brainstorm Topics Template covers:
- Topic 6: Admin & Governance — roles, permissions, governance capabilities
- Topic 8: User Data Management — end-users managing their own profile/account data

**Gap identified:** There is no topic covering how the application's core **business entities** (DB
tables / domain objects) are managed by admins via CRUD interfaces. For example: managing products,
orders, categories, blog posts, organizations, tenants, etc. — the day-to-day operational data
that admins need to create, read, update, and delete from the admin panel.

This is a distinct concern from governance (who can do what) and from user data (users editing
their own profile). It is the **admin-facing data management layer** for all other entities in the DB.

## Details

### What's missing — proposed new brainstorm Topic

**Topic N: Admin Entity Management (ENH-068)**

Core questions to add to the brainstorm Topics Template:

1. **Which DB entities need admin CRUD interfaces?**
   - List all tables/domain objects (e.g., products, orders, categories, tenants, campaigns…)
   - Priority: which are must-have admin UI vs. manageable via seed/migration/API only?

2. **List view requirements per entity**
   - Columns displayed; default sort; pagination (page size, cursor vs. offset)
   - Search/filter fields; range filters (date, price); multi-select filters
   - Bulk actions (bulk delete, bulk status change, bulk export)

3. **Create/Edit form requirements**
   - Required vs. optional fields; validation rules
   - Rich text / media upload fields; nested entity relations (e.g., product → category)
   - Inline vs. separate page; auto-save / draft mode

4. **Delete semantics**
   - Hard delete vs. soft delete (is_deleted / deleted_at flag)
   - Cascade rules — what happens to related records?
   - Restore / undelete capability needed?

5. **Import / Export**
   - CSV/XLSX import for bulk data seeding (e.g., product catalog import)
   - Export filtered data to CSV/XLSX/JSON from list views
   - Import validation (error report for malformed rows)

6. **Audit trail for data changes**
   - Which entities require change history? (actor + timestamp + diff)
   - Separate audit log table vs. event-sourced approach
   - Surfaced in admin UI or log-only?

7. **Admin-to-entity ownership / scoping**
   - Are some entities scoped per tenant/org (multi-tenant)?
   - Can org_admin manage only their own org's entities vs. super_admin sees all?

8. **Read-only vs. editable entities**
   - Some entities generated by the system (e.g., transaction records) — read-only in admin
   - Which entities are user-generated and which are system-generated?

## Discussion

The gap is clear: a user brainstorming a SaaS product will discuss their ERD (entities) and
admin roles (Topic 6) but never explicitly be prompted to think through HOW those entities
are administered day-to-day — the CRUD screens, list/filter/bulk behaviors, import/export,
and soft-delete semantics. Without this, the admin panel scope is often underspecified and
surfaces as rework during implementation.

The topic also naturally cross-references the ERD page (ENH-027): if an ERD is defined, the
entity list there should be the starting point for the admin entity management CRUD coverage.
