## Summary

When `vp-crystallize` runs (or on a new `vp-crystallize --context-files` sub-command), ViePilot
should generate the **adapter-specific AI context file** for the active adapter. Each AI coding
tool has a canonical file it reads for persistent project-level instructions — ViePilot should
produce all of them so the project works seamlessly regardless of which adapter the developer uses.

## Background

Currently `vp-crystallize` generates `.viepilot/AI-GUIDE.md`, `PROJECT-CONTEXT.md`,
`SYSTEM-RULES.md`, etc. — these are **multi-reader, multi-adapter docs**. None of the 5
supported adapters have their native context file generated:

| Adapter | Native Context File | Format | Read by |
|---------|---------------------|--------|---------|
| Claude Code | `CLAUDE.md` | Markdown, any depth | Auto-loaded at session start |
| Cursor | `.cursor/rules/*.mdc` (new) or `.cursorrules` (legacy) | MDC frontmatter + Markdown | Auto-applied per rule scope |
| Codex CLI | `AGENTS.md` | Markdown | Loaded as system context |
| Antigravity (Gemini CLI) | `GEMINI.md` | Markdown | Auto-loaded at session start |
| GitHub Copilot | `.github/copilot-instructions.md` | Markdown | Applied to all Copilot sessions |

Without these files, each adapter "starts blind" on a project. With them, the AI already knows
the project's architecture, coding standards, and ViePilot workflow conventions from turn 1.

## Feature Design

### Option A: Integrate into existing crystallize flow (Recommended)

Add a new step to `workflows/crystallize.md` (e.g. **Step 1F: Generate Adapter Context Files**):

```
Step 1F: Adapter Context Files (NEW)
→ Detect active adapter (adapter-context.cjs)
→ Generate native context file for that adapter
→ Optionally: --all-adapters flag to generate all 5 at once
```

### Option B: Standalone sub-command

```
/vp-crystallize --context-files        → generate for detected adapter only
/vp-crystallize --context-files --all  → generate all 5
vp-tools context-files                 → CLI equivalent
```

Both options can coexist — integrated in step, also callable standalone.

### Content Strategy

Each file is a **condensed, adapter-formatted distillation** of the existing `.viepilot/` docs:

**Source mapping:**
```
CLAUDE.md / GEMINI.md / AGENTS.md
  ← .viepilot/AI-GUIDE.md (nav + architecture summary)
  ← .viepilot/SYSTEM-RULES.md (coding standards, gates, patterns)
  ← .viepilot/PROJECT-CONTEXT.md (domain knowledge, business rules)
  ← Active adapter section from skills (which tools are available)

.cursor/rules/viepilot-context.mdc
  ← Same sources but wrapped in MDC frontmatter:
    globs: ["**/*"] for global rules
    alwaysApply: true
  ← Cursor supports multiple rule files → split into:
    .cursor/rules/00-architecture.mdc
    .cursor/rules/01-coding-standards.mdc
    .cursor/rules/02-viepilot-workflow.mdc

.github/copilot-instructions.md
  ← Same sources, compressed (Copilot has shorter context window)
```

### Generated File: `CLAUDE.md` (Claude Code)

```markdown
# {PROJECT_NAME} — AI Context

## Architecture
{from ARCHITECTURE.md summary — system components, data flow}

## Coding Standards
{from SYSTEM-RULES.md — naming, patterns, do/don't}

## ViePilot Workflow
- Run `/vp-auto` to execute planned phases
- Run `/vp-request` to log bugs or feature requests
- Phase state: `.viepilot/phases/*/PHASE-STATE.md`
- Current tracker: `.viepilot/TRACKER.md`

## Key Domains
{from PROJECT-CONTEXT.md — business domain, entities, rules}

## Stack
{from STACKS.md — tech stack, versions, key libs}
```

### Generated File: `.cursor/rules/00-viepilot.mdc` (Cursor)

```
---
description: ViePilot project context and coding standards
globs: ["**/*"]
alwaysApply: true
---

# {PROJECT_NAME} Context
{same content as CLAUDE.md, cursor-formatted}
```

### Generated File: `AGENTS.md` (Codex CLI)

```markdown
# {PROJECT_NAME} Agent Instructions

{same core content, Codex-optimized — emphasize apply_patch patterns,
no WebSearch, no interactive prompts, patch-first approach}
```

### Generated File: `GEMINI.md` (Antigravity)

```markdown
# {PROJECT_NAME} Context
{same core content — shell-first approach, no Agent spawning}
```

### Generated File: `.github/copilot-instructions.md` (GitHub Copilot)

```markdown
# Copilot Instructions for {PROJECT_NAME}
{compressed core — Copilot context is shorter, focus on coding standards}
```

### Update behavior

On re-run:
- Files are **overwritten** (content should always reflect latest `.viepilot/` state)
- Track which files were generated in `.viepilot/PROJECT-META.md` under `## Context Files`
- Warn if `.viepilot/AI-GUIDE.md` hasn't been crystallized yet

## Technical Notes

### Generation approach

Content generation uses the same LLM-driven approach as other vp-auto tasks:
- Read source `.viepilot/` files
- Condense + reformat for target adapter
- Write to correct path

NOT a static template system — the content is dynamically distilled so it stays concise and
relevant to the actual project.

### Cursor MDC format (research)

Cursor's new rules format (`.cursor/rules/*.mdc`) uses frontmatter:
```
---
description: Short description shown in rules picker
globs: ["src/**/*.ts", "**/*.tsx"]   # scope by file pattern
alwaysApply: true                     # or false for on-demand
---
```
Legacy `.cursorrules` (flat file) is also supported for compatibility.
Recommendation: generate BOTH `.cursorrules` (compat) + `.cursor/rules/` (new).
