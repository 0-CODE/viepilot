# Enhancement: Workflow Upgrade Awareness — Brainstorm Gap Detection + Crystallize Re-scan

## Summary

When ViePilot is updated, existing brainstorm sessions and crystallize artifacts may be missing
coverage that new workflow versions introduce (e.g., Topic 6 Admin, Topic 7 Content, Topic 8 User Data).
Two complementary mechanisms are needed:

1. **`vp-brainstorm` gap detection** — on re-invocation, auto-reads the previous session file,
   compares covered topics against the current workflow's Topics Template, and proactively proposes
   supplementary discussion for any topics the old session never addressed.

2. **`vp-crystallize` re-scan mode** — detects version delta between when the project was last
   crystallized and the current workflow version; scans existing `.viepilot/` artifacts for gaps
   introduced by new workflow features; proposes targeted patch updates or full re-generation.

## Details

### Part A — `vp-brainstorm` Upgrade Gap Detection

**Trigger:** User re-invokes `vp-brainstorm --continue` (or opens an existing session) on a
ViePilot install that is newer than the session's recorded `workflow_version`.

**Behavior:**
1. Read `docs/brainstorm/session-*.md` — extract `workflow_version` from session header (or
   infer from which topics are present).
2. Compare against current `workflows/brainstorm.md` Topics Template to detect missing topics
   (e.g., session has topics 1–5 but current workflow has topics 1–9).
3. Show a proactive upgrade banner:
   ```
   🔄 Upgrade gap detected — this session was created with an older workflow version.
   Missing topics: Topic 6 (Admin & Governance), Topic 7 (Content Management),
                   Topic 8 (User Data Management)
   Discuss these gaps now? [Yes — add to session | Skip | Remind me at /save]
   ```
4. If user agrees: run the missing topic Q&A blocks inline; append results to the existing
   session file under a `## Upgrade supplement (vX.Y.Z → vA.B.C)` section.
5. Session file records `upgrade_supplement_version` for idempotency (don't re-surface
   already-supplemented topics on next open).

**Session header addition:**
```yaml
workflow_version: 2.34.0
upgrade_supplement_version: 2.34.0   # set after supplement is completed
```

### Part B — `vp-crystallize` Re-scan / Upgrade Mode

**Trigger:** User re-invokes `/vp-crystallize --upgrade` (or crystallize auto-detects that
`.viepilot/` artifacts were generated with an older workflow version and current session has
upgrade supplement sections).

**Behavior:**
1. Read `.viepilot/PROJECT-CONTEXT.md` header — extract `crystallize_version` (added by this
   feature).
2. Compute delta: which new Step 1D export items exist in current workflow that weren't present
   at `crystallize_version` (e.g., `## Admin & Governance`, `## Content Management`,
   `## User Data Management`).
3. For each gap item, check if the session has the corresponding YAML section
   (`## admin`, `## content`, `## user_data`) — if missing, suggest running
   `vp-brainstorm` gap detection first (soft gate, non-blocking).
4. Present upgrade menu:
   ```
   🔄 Crystallize upgrade available (v2.20.0 → v2.34.0)
   Detected gaps in existing artifacts:
     • ## Admin & Governance — not present in PROJECT-CONTEXT.md
     • ## Content Management — not present
     • ## User Data Management — not present
   Options:
     1. Patch — append missing sections only (non-destructive)
     2. Full re-generate — regenerate all .viepilot/ artifacts from scratch
     3. Skip — continue with current artifacts
   ```
5. **Patch mode:** reads architect `notes.md` YAML sections → runs only the missing Step 1D
   export items → appends to existing `PROJECT-CONTEXT.md`, `ARCHITECTURE.md` etc. without
   touching other content. Commits with message `chore: crystallize patch vX.Y.Z → vA.B.C`.
6. **Full re-generate mode:** runs full crystallize flow from Step 0 using existing session +
   architect artifacts as source; overwrites `.viepilot/` (backup first to
   `.viepilot/backup-pre-regen-{timestamp}/`). Commits.
7. Updates `crystallize_version` field in generated artifact headers.

### Part C — Version Metadata in Session + Artifact Headers

Both features rely on version stamps:

- **Brainstorm session file header** — add `workflow_version: {semver}` when session is
  created or last saved.
- **PROJECT-CONTEXT.md header comment** — add `<!-- crystallize_version: {semver} -->` or
  YAML frontmatter when file is written by crystallize.
- **`vp-tools info`** — expose `workflow_version` (already exists as `version` in
  `package.json`); used by both detection paths.

## Discussion

User motivation: as ViePilot grows new Topics (Admin, Content, User Data) and new crystallize
export steps, existing projects created before these additions silently miss the coverage.
There is currently no mechanism for users to discover or fill these gaps without manually
re-running brainstorm from scratch. This feature closes the upgrade continuity loop.

Two delivery options:
- **Phased:** brainstorm gap detection first (lower risk), crystallize re-scan second
- **Bundled:** both in one phase (cohesive UX — user upgrades brainstorm → then crystallize
  picks up the supplement automatically)
