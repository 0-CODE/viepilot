# BUG: vp-auto ignores scaffold/create-project best practice — handcrafts files instead

## Meta
- **ID**: BUG-020
- **Type**: Bug
- **Status**: done (Phase 102 → v2.37.0)
- **Priority**: critical
- **Created**: 2026-04-22
- **Reporter**: User
- **Assignee**: AI

## Summary
When `vp-auto` executes project setup tasks for framework-based projects (Laravel, Rails, Next.js, etc.), it **ignores the canonical scaffold/create-project command** and instead manually creates individual files and directories from scratch. This produces incorrect project structure, missing framework internals, and unmaintainable bootstrapped code.

## Details

### Actual Behavior
For a Laravel project, `vp-auto` creates files manually:
```
mkdir -p app/Http/Controllers
touch composer.json
touch app/Http/Controllers/UserController.php
# ... etc
```
Instead of running the correct scaffold command:
```
composer create-project laravel/laravel my-project
```

### Expected Behavior
For **any framework stack**, `vp-auto` MUST:
1. **Detect** that the first task is a project setup task (via stack preflight or task keywords)
2. **Run the canonical scaffold command** first (e.g. `composer create-project`, `npx create-next-app`, `rails new`, `django-admin startproject`, `nest new`, etc.)
3. **Only after scaffold succeeds**, continue with custom configuration tasks
4. **Never handcraft** files that the scaffold command would have generated

### Root Cause
`workflows/autonomous.md` has a "Stack Preflight" step that reads `STACKS.md` and `~/.viepilot/stacks/{stack}/SUMMARY.md`, but there is **no scaffold-first enforcement gate**. The workflow does not:
- Check if `SUMMARY.md` defines an `init_command` / `scaffold_command`
- Require the scaffold command to be run before any file creation
- Block handcrafting files that belong to the framework scaffold

### Affected Stacks (examples)
| Stack | Correct scaffold command |
|-------|--------------------------|
| Laravel | `composer create-project laravel/laravel {name}` |
| Next.js | `npx create-next-app@latest {name}` |
| Ruby on Rails | `rails new {name}` |
| Django | `django-admin startproject {name}` |
| NestJS | `npx @nestjs/cli new {name}` |
| React | `npx create-react-app {name}` |
| Vue | `npm create vue@latest` |
| Spring Boot | Spring Initializr / `spring init` |

### Affected Files
- `workflows/autonomous.md` — missing scaffold-first gate in stack preflight step
- `~/.viepilot/stacks/{stack}/SUMMARY.md` files — need `scaffold_command` / `init_command` field
- `workflows/brainstorm.md` — brainstorm should elicit scaffold approach when stack is identified
- `workflows/crystallize.md` — PROJECT-CONTEXT.md should capture `scaffold_command` per stack

## Acceptance Criteria
- [ ] `workflows/autonomous.md` stack preflight checks for `scaffold_command` field in stack SUMMARY.md
- [ ] If `scaffold_command` exists and project directory does not yet exist: run scaffold command BEFORE any other file operations in that phase
- [ ] Task contract validation blocks tasks that create files scaffold would generate, unless scaffold has already been run
- [ ] `~/.viepilot/stacks/*/SUMMARY.md` includes `scaffold_command` field for framework stacks
- [ ] Unit test verifies the scaffold-first gate logic is documented in autonomous.md
- [ ] Handcraft prevention warning shown when agent is about to create framework-native files without scaffold

## Discussion
This is a **correctness** bug with wide impact: any user who runs vp-auto on a new Laravel/Rails/Next.js/etc. project will get a broken project structure that:
1. Misses framework-generated files (bootstrap files, service providers, config stubs)
2. Has wrong directory layout
3. May not be runnable at all
4. Cannot be fixed incrementally — requires starting over with the correct scaffold

The fix requires a scaffold-first gate in `autonomous.md` that is enforced at the stack preflight step, plus each stack's SUMMARY.md needing a `scaffold_command` field.
