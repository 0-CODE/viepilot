# ENH-033: Architect HTML — Item IDs + Approve/Edit prompt-copy buttons

## Summary
Each section/item in all Architect HTML workspace pages (12 templates) should receive a unique
stable **ID** and two action buttons — **Approve** and **Edit** — that copy a pre-formatted
agent prompt to the clipboard. This enables precise, identity-tagged communication between the
HTML workspace and the AI agent, eliminating ambiguity when reviewing or editing architect artifacts.

## Problem Being Solved
When an AI agent generates or updates architect content (decisions, ERD entities, API endpoints,
use cases, etc.) across 12 HTML pages, there is no way to reference a specific item precisely in
follow-up chat. The user must describe the item in natural language, which leads to misidentification
or incomplete updates. With stable IDs and copy-prompt buttons, the agent receives a deterministic
handle (`[ARCH-decisions-D3]`, `[ARCH-erd-E1]`, etc.) and can locate/update the exact item.

## Desired Behavior

### 1. Item IDs
- Every renderable item/entry/card/row in each page gets a unique ID scoped to the page.
- ID format: `{page-slug}-{type}{index}` — e.g. `decisions-D1`, `erd-E3`, `apis-A2`, `use-cases-UC4`
- ID displayed as a small badge next to the item title (not intrusive).

### 2. Approve Button
- Button label: ✅ Approve (or green checkmark icon)
- On click: copies to clipboard a prompt like:
  ```
  [ARCH:decisions-D1] APPROVE — "{item title}" is confirmed as-is. No changes needed.
  ```
- Visual feedback: button briefly shows "Copied!"

### 3. Edit Button
- Button label: ✏️ Edit (or pencil icon)
- On click: copies to clipboard a prompt like:
  ```
  [ARCH:decisions-D1] EDIT — "{item title}": I want to change/update this item. Current content: "{brief excerpt}". Please ask me what to change.
  ```
- The copied prompt is a conversation starter — the agent reads it, acknowledges the item by ID,
  then asks clarifying questions to perform the edit.
- Visual feedback: button briefly shows "Copied!"

### 4. Pages in scope (all 12 templates)
- `decisions.html` — each decision card
- `architecture.html` — each component/service block
- `data-flow.html` — each flow step or node description
- `tech-stack.html` — each technology entry
- `tech-notes.html` — each note/section
- `feature-map.html` — each feature entry
- `erd.html` — each entity
- `user-use-cases.html` — each use case row
- `sequence-diagram.html` — each scenario block
- `deployment.html` — each environment/infra entry
- `apis.html` — each endpoint row
- `index.html` — no items (hub nav page, skip)

### 5. Style
- Buttons are small (compact), appear inline near item title or in item action area.
- Use existing `style.css` color palette; do not break current layout.
- Buttons hidden by default, shown on hover over the item (CSS `group-hover` pattern or similar).

## Isolation Rule (CRITICAL)

**Approve and Edit are strictly per-item, per-page. There is NO cascading between pages.**

- Approving `[ARCH:architecture-C2]` (a component in architecture.html) does **NOT** approve
  related entities in `erd.html`, endpoints in `apis.html`, or any other page.
- Approving `[ARCH:decisions-D1]` does **NOT** imply any other decision or item is approved.
- Each item lives in its own approval namespace: `{page-slug}:{item-id}`.
- The agent must treat each prompt as scoped exclusively to the identified item on the identified page.
  Any cross-page impact (e.g. "this decision affects the ERD") must be a **separate, explicit** follow-up
  — the agent must never infer approval of related items.

**Why this matters:**
Different pages represent different artifact types (architecture topology, data schema, API contracts, etc.)
A component being approved in one domain does not mean its counterpart in another domain is ready.
Cascading approval would cause premature confirmation of items the user has not reviewed.

**Prompt format enforces isolation:**
```
[ARCH:{page-slug}:{item-id}] APPROVE — "{title}" on {page-slug} page only.
[ARCH:{page-slug}:{item-id}] EDIT — "{title}" on {page-slug} page. Current: "{excerpt}". What should change?
```

## Acceptance Criteria

- [ ] Edit button copies correct prompt format with item ID + page slug + title + excerpt
- [ ] Both buttons show "Copied!" feedback for 1.5s then revert
- [ ] IDs are stable — re-rendering same data produces same IDs (index-based)
- [ ] Prompt text explicitly scopes to page — no cross-page language
- [ ] Agent-facing spec in `workflows/brainstorm.md` documents the isolation rule
- [ ] Buttons use existing style.css variables; no new external CSS dependencies
- [ ] style.css updated with button styles if needed
- [ ] No JS framework dependencies added — vanilla JS only

## Implementation Notes
- Buttons can be injected via the same JS that renders item lists, or via a shared helper
  function `makeActionButtons(pageSlug, itemId, title, excerpt)` in a shared script.
- Copy-to-clipboard: use `navigator.clipboard.writeText()` with fallback `document.execCommand('copy')`.
- Consider a shared `architect-actions.js` or inline the logic in each template's existing script.
- Prompt format strings should be defined as constants so they can be updated globally.
- **The isolation rule must also be documented in `workflows/brainstorm.md`** under the Architect
  Design Mode section so the agent knows not to cascade approvals when processing copied prompts.
