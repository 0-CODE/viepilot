## Summary
The `.docx` output of `/vp-proposal` should be AI-native — generated independently by an AI Agent with deep, contextual content — not derived from the slide manifest. It should include UML/architecture diagrams (Mermaid or PlantUML-style) embedded as images or described as structured text blocks, providing technical depth far beyond what slides convey.

## Details

### Current behavior
- `workflows/proposal.md` Step 7 generates `.docx` by expanding the slide manifest (same content as pptx, slightly longer)
- Document sections use `{{placeholder}}` text that the AI fills in — but the source of truth is still the slide-level manifest
- No diagrams; no independent deep-content generation pass

### Desired behavior

**1. Separate AI generation pass for docx**
- After the slide manifest is generated (Step 4), trigger a **dedicated docx content generation** step
- AI Agent receives: brainstorm session + quality brief meta + proposal type → generates full document content independently
- Output: structured JSON with per-section rich content (paragraphs, lists, table data, diagram specs)
- This content is richer and longer than slides — not a slide expansion

**2. UML / Architecture diagram support**
Supported diagram types (auto-selected based on proposal type):
- `flowchart` — process flow, user journey (always relevant)
- `sequenceDiagram` — API interactions, system sequences (tech-architecture, product-pitch)
- `classDiagram` — data model, entity relationships (tech-architecture)
- `erDiagram` — database schema (tech-architecture with data layer)
- `gantt` — project timeline in diagram form (project-proposal, general)

Rendering approach:
- Option A: Embed Mermaid source as code blocks in `.md` companion file (renderable in GitHub/Notion)
- Option B: Use `mermaid-js` to render SVG → embed as image in `.docx` (requires headless Chromium or `@mermaid-js/mermaid-cli`)
- Option C: Describe diagrams as structured text tables in `.docx` (no render dep, simpler)
- **Recommended: Option A (Mermaid in .md) + Option C (table fallback in .docx)** — no new binary deps

**3. Document content spec per section**
Each section gets AI-generated content at a deeper level than slides:
- Executive Summary: 3–5 paragraphs (problem → solution → benefits → recommendation)
- Technical Architecture: architecture narrative + component table + sequence description
- Data Model: entity list + relationships (if relevant)
- Timeline: Gantt-style narrative + diagram spec
- Risk Register: table (Risk | Probability | Impact | Mitigation)
- Glossary: key terms defined (for non-technical audience)

**4. Workflow change**
Add `<step name="generate_docx_content">` between Step 4 (manifest) and Step 7 (generate_docx):
```
Step 4b: AI Docx Content Generation
- Input: brainstorm session + quality brief meta + proposal type + manifest
- Output: docxContent JSON with per-section content + diagram specs
- This is a separate AI call with a longer, deeper prompt
- diagrams[] array: [{type, title, mermaidSource}]
```

## Discussion
This separates the docx from being a "slide expansion" into being a proper technical document that stands alone. The Mermaid-in-md approach is zero-dependency and works everywhere (GitHub, VS Code, Notion, Obsidian). The `.docx` gets structured table fallbacks so it's still useful without a Mermaid renderer.

The AI call for docx content should be treated as a second, deeper generation pass — different prompt, different output schema. It's an independent artifact, not a derivative of the slides.
