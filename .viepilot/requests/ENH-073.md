## Summary
A new `/vp-persona` skill that creates a **user-level, cross-project identity** for ViePilot.
Allows multi-domain professionals ("kiêm nhiệm") to customize ALL ViePilot workflows — brainstorm topics,
phase templates, stack best practices, architect pages — to match their domain, profession, and interests.
No re-declaring context per project. Supports multiple named personas with switch + merge.

## Current Behavior
- ViePilot has no user-level identity — only `language.*` in `~/.viepilot/config.json`
- Global profiles (FEAT-009) are project-specific, not cross-project
- Brainstorm topics are general (Admin, Content, User Data) — irrelevant for embedded/data-science/devops
- Phase templates are hardcoded — wrong for ML pipeline, firmware, mobile
- Stack best practices must be re-declared per project
- Multi-domain professionals must manually set context every time

## Desired Behavior

### `/vp-persona` skill with wizard
5-question setup wizard (domain, role, stacks, team size, output style).
Saves to `~/.viepilot/personas/{name}.json`. Active persona read by all vp-* skills.

### Multi-persona architecture
```
~/.viepilot/
  persona.json            ← active persona (pointer)
  personas/
    web-saas.json         ← named persona 1
    embedded-iot.json     ← named persona 2
    data-science.json     ← named persona 3
  context-map.json        ← directory → persona bindings
```

### Layered resolution (3 layers)
```
Priority: project > auto-map > global
1. .viepilot/persona-override.json   ← per-project explicit
2. context-map.json pattern match    ← directory-based auto
3. ~/.viepilot/persona.json (active) ← global default
```

### Auto-detection from project files
| Signal file | Detected domain |
|------------|-----------------|
| `package.json` + `prisma/` | web-saas |
| `CMakeLists.txt` + `sdkconfig` | embedded |
| `requirements.txt` + `notebooks/` | data-science |
| `pubspec.yaml` / `*.xcodeproj` | mobile |
| `Dockerfile` + `terraform/` | devops |
| `pyproject.toml` + `langchain` | ai-product |

### Persona schema
```json
{
  "name": "Web SaaS Dev",
  "domain": "web-saas",
  "role": "full-stack",
  "stacks": ["nextjs", "nestjs", "postgresql"],
  "team_size": "solo",
  "output_style": "lean",
  "brainstorm": {
    "topic_priority": ["auth", "user-data", "api", "admin"],
    "topic_skip": []
  },
  "phase_template": "lean-startup",
  "interests": ["ai-integration", "saas"]
}
```

### persona-context.md injection
All vp-* skills read `persona-context.md` and inject it as context. Example:
```
## User Persona
- Role: Full-stack developer, solo founder
- Domain: Web SaaS, AI-integrated products
- Preferred stacks: Next.js / NestJS / PostgreSQL
- Output style: lean
- Phase template: lean-startup
```

### Switch + Merge
- **Switch**: `/vp-persona --switch embedded` — instant context switch
- **Merge**: `/vp-persona --merge web-saas embedded-iot` → creates hybrid persona (e.g. IoT Dashboard project)
  - Topics: union of both domains
  - Phase template: hybrid (Board Bring-Up + Web Scaffold → MQTT Bridge + REST API → Dashboard UI + OTA)
  - Architect pages: union of both

### Workflow adaptations
- **vp-brainstorm**: topics reordered by `topic_priority`; `topic_skip` topics hidden; domain pack topics injected; session-start prompt for multi-domain users
- **vp-auto**: stack preflight uses persona stacks; `persona-context.md` injected per task
- **vp-crystallize**: `output_style: lean` → concise docs; `enterprise` → full docs + compliance sections
- **vp-evolve**: new phase suggestions follow `phase_template`

### `/vp-persona` commands
```
/vp-persona                      → Show active persona + list
/vp-persona --create <name>      → Wizard to create new named persona
/vp-persona --switch <name>      → Set active persona
/vp-persona --list               → List all saved personas
/vp-persona --bind . <name>      → Bind current project dir to persona
/vp-persona --auto               → Enable auto-detect from project files
/vp-persona --merge <a> <b>      → Create hybrid persona from two
/vp-persona --edit               → Edit active persona fields
/vp-persona --export / --import  → Backup/share/onboard
/vp-persona --reset              → Clear all personas
/vp-persona --calibrate          → Analyze traces + propose persona/overlay updates
/vp-persona --calibrate --auto   → Apply low-risk changes automatically
```

### Phase 5 — Adaptive Persona Calibration (AI-driven self-optimization)

Agents (vp-auto, vp-brainstorm) observe usage patterns and propose persona + workflow overlay
updates. Based on Reflexion pattern (arxiv 2303.11366) and JSON Patch RFC 6902.

#### Session Trace Logging (passive, always-on)
After each session, async write to `~/.viepilot/traces/{session-id}.json`:
```json
{
  "skill": "vp-brainstorm", "persona": "web-saas",
  "topics_offered": ["auth","content-mgmt","admin","user-data"],
  "topics_discussed": ["auth","admin","user-data"],
  "topics_skipped": ["content-mgmt"],
  "stacks_mentioned": ["nextjs","postgresql","stripe"],
  "duration_min": 42
}
```

#### Reflexion Memory (`~/.viepilot/persona-reflections.json`)
Pattern detection across N traces (≥3 signals = proposal threshold):
- Topic skip_rate ≥ 0.70 → propose adding to `topic_skip`
- Stack mentioned in ≥ 0.80 of sessions → propose adding to persona stacks
- Phase always added manually → propose adding to `phase_template`
- Rejected proposals stored in `guardrail_journal` → never re-proposed

#### Workflow Overlays (`~/.viepilot/overlays/{persona}/`)
JSON Patch RFC 6902 overlays — base workflow files never modified:
```json
// ~/.viepilot/overlays/web-saas/brainstorm.patch.json
[
  { "op": "skip_step", "step_id": "topic_content_management",
    "reason": "skipped in 4/5 sessions" },
  { "op": "set_default", "step_id": "stack_selection",
    "value": ["nextjs","postgresql","stripe"] }
]
```
ViePilot upgrades update base workflows; overlays compose on top — non-destructive.

#### Risk-tiered approval gates
| Risk | Examples | Behavior |
|------|----------|---------|
| 🟢 Low | Add stack, reorder topic priority | Auto-apply + notify |
| 🟡 Medium | Add to topic_skip, set phase default | 1-click approve prompt |
| 🔴 High | Modify overlay logic, restructure phase template | Explicit review + confirm |

#### `vp-calibrate` flow
1. Read all installed workflow files → build step inventory with skip_rate per step
2. Read N recent traces from `~/.viepilot/traces/`
3. Detect patterns via skip rate + stack frequency + phase patterns
4. Generate proposals grouped by risk tier
5. Present 🟡/🔴 proposals via AUQ; auto-apply 🟢
6. Write approved changes → `persona.json` updates + overlay patch files
7. Log rejections to `guardrail_journal` (never re-propose same pattern)
