## Summary
When `vp-crystallize --brownfield` is used, the workflow says "skip Step 1, continue from
Step 0 → Step 2 onward" but does NOT explicitly state whether sub-steps 1A, 1B, 1C, 1D
are skipped too. This creates ambiguity at runtime.

## Details

### Current behavior
- `crystallize.md` Step 0-B: brownfield scanner runs → stub file created
- Step 0-C: `session-brownfield-import.md` generated
- Step 1 comment (line ~153): "Skip Step 1 → continue Step 2 onward"
- **But Steps 1A–1D are sub-steps of Step 1:**
  - 1A: UI direction gate
  - 1B: Stack research cache
  - 1C: Architect artifact consumption
  - 1D: auto-activate suggestion

### Ambiguity
Step 1's `analyze_brainstorm` logic iterates "for each session file" — which now includes
`session-brownfield-import.md`. So Step 1 is **not** truly skipped — it reads the brownfield
stub. But 1A–1D are unclear:
- 1A (UI direction): a brownfield project may have UI, should this check run?
- 1B (stack research): brownfield scanner already detected stacks — re-run needed?
- 1C (architect artifacts): only relevant if architect HTML workspace exists
- 1D (auto-activate): only relevant if new feature scope is brainstormed

### Impact
AI agents running `vp-auto` after crystallize may apply wrong sub-steps, causing either
missed stack research (skip 1B) or redundant re-scanning.

## Acceptance Criteria
- [ ] `crystallize.md` has an explicit "Brownfield Execution Path" table showing each step/sub-step: RUN / SKIP / CONDITIONAL
- [ ] 1A: CONDITIONAL (run if `.viepilot/ui-direction/` pre-exists)
- [ ] 1B: CONDITIONAL (skip if brownfield scanner already populated stack cache)
- [ ] 1C: SKIP (no architect workspace in brownfield import)
- [ ] 1D: SKIP (no brainstorm session in brownfield)
- [ ] Contract test covering brownfield path execution gate
