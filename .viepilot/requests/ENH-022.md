## Summary

`crystallize` bỏ sót core CRUD management service phases khi project có domain entities được nêu trong brainstorm. Root cause: không có explicit domain entity extraction step — AI inference bỏ qua CRUD layer, chỉ gen phases cho "technically interesting" signals (streaming, event-driven, etc).

## Observed Gap (Reference: smart-track-platform)

Brainstorm 2026-03-30 nêu rõ các domain entities: tenants, users, vehicles, devices, vehicle_type, protocol_nodes, drivers, fleets.

ROADMAP phases 01-13 kết quả: có Phase 03 (DB migrations) nhưng **không có** phase nào cho management service APIs (TenantService, UserService, VehicleService, DeviceService, FleetService).

Phase 04 task 4.3: "Enrichment: resolve tenant/device" → calls service layer that doesn't exist because no management service phase was generated.

## Root Cause

crystallize.md không có explicit step:
1. Extract domain entities từ brainstorm (tường minh, không inference)
2. Map each entity → required management service phase (CRUD API)
3. Cross-check: task descriptions có "resolve/create/update/manage {entity}" → verify service phase exists

Pattern: crystallize reads brainstorm → extracts streaming/async architecture prominently → generates event/ingest phases → CRUD management layer bị skip vì "implied by DB migrations".

## Scope of Fix

### Fix A — Domain entity checklist step (crystallize.md)

Thêm explicit step sau brainstorm analysis và trước phase generation:

```
Domain Entity Extraction (mandatory):
1. Scan brainstorm for all named entities (nouns that are persisted objects):
   - List each entity: {name}, {type: core|reference|junction}, {needs_crud_api: yes/no}
2. For each entity where needs_crud_api = yes:
   - Check if a management service phase exists in planned phases
   - If not → add phase stub: "{EntityName} Management Service — CRUD API"
3. Output: Entity manifest table (entity, type, service_phase or "MISSING")
```

### Fix C — Dependency validation step (crystallize.md)

Thêm cross-check sau khi gen tất cả phases:

```
Dependency Validation (mandatory):
Scan all task descriptions for patterns:
  - "resolve {entity}", "create {entity}", "update {entity}", "manage {entity}", "lookup {entity}"
For each match:
  - Verify a {EntityName}Service phase exists with lower phase number
  - If not found → WARNING: "Task X.Y depends on {entity} management but no service phase found"
  - Offer: auto-add stub phase OR user confirms intentional (e.g., entity resolved from config)
```
