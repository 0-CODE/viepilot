## Summary
Add a `browser-intake-agent` Claude Code native agent (and companion vp-intake integration) that can fetch and extract ticket/feature-request content from publicly shared web URLs — Google Sheets share links, GitHub Issues, Jira public tickets, Trello public boards, Notion pages — so that the intake pipeline can handle any HTTP(S) public link as a channel source, not just local files or Graph API.

## Problem Statement
Current intake channels are limited to:
- Local CSV / XLSX files (`excel`, `csv` channels)
- SharePoint / Microsoft 365 (via Graph API — requires credentials)
- Google Sheets API (requires OAuth)

Public sharing links (e.g. `https://docs.google.com/spreadsheets/d/xxx/edit?usp=sharing`, GitHub issue URLs, Notion page URLs) are **not supported**. `WebFetch` handles static HTML only and fails on JS-rendered content (Google Sheets, Jira, Trello, Notion are all React/Vue SPAs).

## Proposed Solution

### Core: `browser-intake-agent` (agents/claude-code/)
A new Claude Code native agent powered by **`vercel-labs/agent-browser`** skill — purpose-built for AI agent workflows, ref-based deterministic element selection, Rust daemon for speed:

```yaml
name: browser-intake-agent
model: claude-sonnet-4-6
tools: [Bash, Read]
disallowedTools: [Edit, Write, WebSearch]
# Prerequisite: npx skills add vercel-labs/agent-browser
```

**Operations:**
- `read_url` — navigate to public URL, extract structured rows (title, description, status, labels)
- `screenshot` — capture for human review queue
- `detect_type` — identify page type (google-sheets / github-issue / jira / trello / notion / generic-table)

### Integration points in `skills/vp-intake/SKILL.md`
- New channel type: `browser` with `url` field in channels config
- Step 3.5 routing: when channel.type = `browser`, dispatch `Agent({ subagent_type: "browser-intake-agent", op: "read_url", url: channel.url })`
- Works alongside existing `excel-intake-agent` and `sheets-intake-agent` (same dispatch pattern)

### Fallback chain (progressive degradation)
```
1. browser-intake-agent (Playwright MCP) → full JS-rendered content
2. WebFetch → static HTML (fallback if Playwright MCP not installed)
3. Manual paste → user pastes content as text input
```

## Research Findings (2026-05-23)

### Primary Recommendation: `vercel-labs/agent-browser`
Purpose-built for AI agent browser workflows. Key advantages over alternatives:
- **Rust-based native binary** — no Node.js cold starts, daemon architecture eliminates per-command overhead
- **Ref-based element selection** (`@e1`, `@e2`) — deterministic refs designed for LLM instruction, not brittle CSS selectors
- **Claude Code skill** — install via `npx skills add vercel-labs/agent-browser` (no MCP server needed)
- **APIs**: `open`, `snapshot -i`, `click @ref`, `fill @ref "text"`, `screenshot`, `get-dom`, multi-tab, React introspection
- **License**: Apache-2.0, maintained by Vercel Labs

Install: `npx skills add vercel-labs/agent-browser`
Source: https://github.com/vercel-labs/agent-browser

### Fallback options

| Tool | Integration | Best for |
|------|------------|---------|
| **Playwright MCP** | `claude mcp add playwright` — native CC | General JS SPA automation |
| **Stagehand** | TypeScript SDK, act/extract/observe primitives | Repeated workflows with caching |
| **browser-use** | Python framework, Claude Sonnet 4.6+ | Vision-based reasoning tasks |
| **Chrome DevTools MCP** | Direct Chrome control | Debug/inspection use cases |

**Revised implementation priority**: agent-browser (primary) → Playwright MCP (fallback) → WebFetch (static-only fallback).

## Discussion
User primary use case: intake of publicly shared links that don't require credentials.
- Google Sheets: `spreadsheets/d/xxx/edit?usp=sharing` (read-only public share)
- GitHub Issues list: `github.com/org/repo/issues` (public repos)
- Jira: public project boards (Service Desk public portals)
- Notion: `notion.so/xxx` with "anyone with link" permission
- Trello: public boards

Playwright MCP is the best fit because it is:
- Already supported by Claude Code (no custom MCP server to build)
- Accessibility-tree based → structured extraction, not brittle CSS selectors
- Zero API key requirement
- Cross-browser
