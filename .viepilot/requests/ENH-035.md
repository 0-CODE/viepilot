# ENH: Replace hardcoded .cursor/viepilot with {envToolDir} template variable in skill source files

## Meta
- **ID**: ENH-035
- **Type**: Enhancement
- **Status**: new
- **Priority**: high
- **Created**: 2026-04-10
- **Reporter**: User
- **Assignee**: AI

## Summary
Replace all hardcoded `.cursor/viepilot` occurrences in `skills/*/SKILL.md` source files with a canonical template variable `{envToolDir}`. The install step resolves `{envToolDir}` to each adapter's `executionContextBase` (e.g. `.cursor/viepilot`, `.claude/viepilot`, `.antigravity/viepilot`). This eliminates the `pathRewrite` mechanism and makes the source files platform-neutral.

## Current Behavior (problem)

Source SKILL.md files hardcode the Cursor-first path:
```
<execution_context>
@$HOME/.cursor/viepilot/workflows/autonomous.md
</execution_context>
```

Each new adapter must define `pathRewrite: { from: '.cursor/viepilot', to: '...' }` so the installer does a string-replace at install time. This is fragile (Cursor-biased source, manual from/to per adapter) and looks unprofessional.

**Affected**: 14 of 16 SKILL.md files contain `.cursor/viepilot` (28 occurrences total).

## Desired Behavior

Source SKILL.md files use a neutral placeholder:
```
<execution_context>
@$HOME/{envToolDir}/workflows/autonomous.md
</execution_context>
```

At install time, `viepilot-install.cjs` replaces `{envToolDir}` with `adapter.executionContextBase`:
- Cursor → `.cursor/viepilot`
- Claude Code → `.claude/viepilot`
- Antigravity → `.antigravity/viepilot`
- Any future adapter → just define `executionContextBase`, no `pathRewrite` needed

## Scope

### Part A — Replace in all skill source files (14 SKILL.md files)
Find-replace `$HOME/.cursor/viepilot/` → `$HOME/{envToolDir}/` in `skills/*/SKILL.md`.

Pattern covers both occurrences:
- Inside `<execution_context>@$HOME/.cursor/viepilot/...`
- Inside prose like `` `@$HOME/.cursor/viepilot/...` ``

### Part B — Update `lib/viepilot-install.cjs` install logic
Replace the `pathRewrite` mechanism with template substitution:

```js
// Before (current):
if (adapter.pathRewrite) {
  steps.push({ kind: 'rewrite_paths_in_dir', ..., from: adapter.pathRewrite.from, to: adapter.pathRewrite.to });
}

// After (new):
steps.push({
  kind: 'rewrite_paths_in_dir',
  dir: skillsDir,
  glob: '**/*.md',
  from: '{envToolDir}',
  to: adapter.executionContextBase,   // e.g. '.cursor/viepilot'
});
```

Every adapter now gets the substitution (idempotent for Cursor since `{envToolDir}` → `.cursor/viepilot`).

### Part C — Simplify adapter definitions
- Remove `pathRewrite` field from `claude-code.cjs` (no longer needed)
- Remove `pathRewrite: null` comment from `cursor.cjs`
- Update `antigravity.cjs` (FEAT-014) to not include `pathRewrite`
- Keep `executionContextBase` as the single source of truth for the install path

### Part D — Update adapter index + types (if any)
Remove `pathRewrite` from adapter shape documentation / JSDoc.

### Part E — Contract tests
Update or add tests in `tests/unit/`:
- `vp-adapter-cursor.test.js` — verify `{envToolDir}` in source SKILL.md files (not `.cursor/viepilot`)
- `vp-adapter-claude-code.test.js` — verify no `pathRewrite` field needed / install logic uses `executionContextBase`
- Smoke test: template substitution produces correct path per adapter

### Part F — Update FEAT-014 (Antigravity) request
FEAT-014's `antigravity.cjs` spec should drop `pathRewrite` field — just `executionContextBase: '.antigravity/viepilot'` is enough.
