# ENH-034: vp-brainstorm UI — sync architect HTML templates when gaps/changes are identified

## Summary
When `/vp-brainstorm` (UI mode) identifies changes or gaps related to the architect,
it currently only updates the architect notes (`.viepilot/architect/*/notes.md` or similar)
but does NOT update the architect HTML workspace templates. This makes it hard to
visually observe what changed — the HTML pages are stale while the notes show updates.

## Details
### Current behavior
- `vp-brainstorm` UI mode: when architect changes/gaps are surfaced, only text notes
  are written; architect HTML templates remain unchanged.
- Developer has to manually open HTML files and compare with notes to understand the delta.

### Desired improvement
When `vp-brainstorm` completes a UI session that surfaces architect-related changes or gaps:
1. **Detect affected architect pages** (e.g. ERD, architecture, APIs, use-cases, deployment…)
2. **Propose inline updates** to the relevant architect HTML template(s) — or generate
   a diff/patch the user can review and apply.
3. **Mark affected items** — at minimum, add a visual indicator (e.g. a CSS class or badge)
   to items that have been flagged as "needs review" during brainstorm.

### Why valuable
- Architect HTML workspace is the source-of-truth visual layer; keeping it stale while
  notes diverge undermines the purpose of the Architect Mode (visual observability).
- Reviewers / stakeholders can see changes directly in the HTML pages without reading notes.

### Scope options (for evolve to decide)
- **Option A (minimal):** After brainstorm, add a "⚠️ Needs review" badge/note card to
  affected architect pages so gaps are visible.
- **Option B (full sync):** Parse brainstorm output for architect deltas → update
  relevant `<tr>` or `<div class="card">` content in HTML templates; record what was changed.
- **Option C (guided):** At end of brainstorm, surface a structured "Architect Sync" step
  listing each gap with a prompt the user can run in architect context via the Approve/Edit flow.

## Acceptance Criteria
- [ ] `workflows/brainstorm.md` has a step that detects architect-related gaps from UI brainstorm output
- [ ] Affected architect HTML pages are updated (or flagged) to reflect changes identified in brainstorm
- [ ] Change is traceable: notes + HTML stay in sync after a brainstorm session
- [ ] Existing isolation rule (ENH-033) is respected — per-page, per-item, no cross-page cascade

## Discussion
The core problem: brainstorm lives in markdown / AI context; architect workspace lives in
HTML. There's no automatic bridge between "AI identifies gap" and "HTML template reflects gap".
The Approve/Edit prompt flow (ENH-033) is the right long-term mechanism — Option C
(guided architect sync prompts at end of brainstorm) is most coherent with the existing architecture.
