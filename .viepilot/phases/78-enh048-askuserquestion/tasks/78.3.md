# Task 78.3 — workflows/request.md + workflows/evolve.md

## Objective
Apply the adapter-aware interactive prompt pattern to question-asking blocks in
`workflows/request.md` (type detection, severity/priority selects) and
`workflows/evolve.md` (intent detection, feature info gathering).

## Paths
```
workflows/request.md

```

## workflows/request.md — Sections to update

### 1. Request type detection (Step 2 — detect_type)
The main "What type of request?" menu — highest priority:
- AUQ spec: question "What type of request?" with 4 options:
  - Bug Report (something is broken)
  - Enhancement (improve existing feature)
  - Feature Request (new functionality)
  - Technical Debt (cleanup / refactor)
- multiSelect: false
- Note: "Brainstorm" and "List" are routing actions, not request types → keep as text

### 2. Bug severity selection (Step 4A — gather_bug)
After bug details gathered, the "Severity?" question:
- AUQ spec: question "What is the bug severity?" with 4 options:
  - Critical (system down, data loss)
  - High (major feature broken)
  - Medium (feature impaired)
  - Low (minor cosmetic issue)
- multiSelect: false

### 3. Feature priority selection (Step 4B — gather_feature)
"Priority?" question for feature requests:
- AUQ spec: question "What is the feature priority?" with 3 options:
  - Must-have (critical for release)
  - Should-have (important but not blocking)
  - Nice-to-have (can defer)
- multiSelect: false

### 4. Feature brainstorm routing question (Step 4B)
"Want to brainstorm this in detail? (y/n)":
- AUQ spec: question "Brainstorm this feature in depth?" with 2 options:
  - Yes — start brainstorm session first
  - No — log request directly
- multiSelect: false

### 5. Adapter compatibility note
Add adapter compatibility table near top of file.

## workflows/evolve.md — Sections to update

### 1. Intent detection (Step 2 — ask_intent)
"How would you like to evolve the project?":
- AUQ spec: question "How would you like to evolve the project?" with 3 options:
  - Add Feature (add new capability to current milestone)
  - New Milestone (archive current, start fresh scope)
  - Refactor (improve existing code, no new features)
- multiSelect: false

### 2. Estimated complexity (Step 3A)
"Estimated complexity?" during feature info gathering:
- AUQ spec: question "Estimated implementation complexity?" with 4 options:
  - S — Small (few hours, isolated change)
  - M — Medium (1–2 days, 1–2 files)
  - L — Large (3–5 days, multiple modules)
  - XL — Extra Large (1+ week, architectural change)
- multiSelect: false

### 3. Brainstorm routing (Step 3A)
"Need deep brainstorm?" question:
- AUQ spec: question "Does this feature need a brainstorm session first?" with 2 options:
  - Yes — route to /vp-brainstorm
  - No — proceed to task planning directly
- multiSelect: false

### 4. Adapter compatibility note
Add adapter compatibility table near top of evolve.md.
