# Task 81.4 — Version Bump Unification: SYSTEM-RULES.md + evolve.md + autonomous.md (ENH-053)

## Objective
Consolidate version bump rules from two workflows into a single canonical table in
`SYSTEM-RULES.md`. Both `workflows/evolve.md` and `workflows/autonomous.md` currently
define bump rules independently, creating risk of contradiction in mixed-type phases.

## Paths
- `SYSTEM-RULES.md`
- `workflows/evolve.md`
- `workflows/autonomous.md`

## File-Level Plan

### Edit 1 — SYSTEM-RULES.md: add canonical Version Bump Rules section

Find the existing versioning section (search for "Semantic Versioning" or "version bump")
and add/replace with this canonical table:

```markdown
## Version Bump Rules (Canonical)

All version bump decisions across `/vp-evolve` and `/vp-auto` must follow this table:

| Change type | Bump | Example |
|-------------|------|---------|
| Breaking API/behavior change | **MAJOR** | 2.x.x → 3.0.0 |
| New feature or capability added | **MINOR** | 2.17.x → 2.18.0 |
| Bug fix only (no new features) | **PATCH** | 2.17.0 → 2.17.1 |
| Refactor / internal cleanup (no behavior change) | **PATCH** | 2.17.0 → 2.17.1 |
| Mixed phase (feature + fix) | **MINOR** — highest applicable bump wins | 2.17.x → 2.18.0 |
| Docs / tests only | **PATCH** | 2.17.0 → 2.17.1 |

**Precedence rule:** MAJOR > MINOR > PATCH. When a phase contains multiple change types,
apply the highest-priority bump. A phase with both a new feature and a bug fix → MINOR.

> **Source of truth:** This table is the only definition of bump rules. Do not redefine
> bump logic inline in `evolve.md` or `autonomous.md` — reference this section instead.
```
