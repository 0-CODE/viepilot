## Objective

Refactor `workflows/resume.md` (hoặc tạo nếu chưa có) để implement 3-tier tiered restore:
- Quick (<30min gap): HANDOFF.json position + PHASE-STATE.md current task
- Standard (30min–4h): Full HANDOFF.json + PHASE-STATE.md + current TASK.md
- Full (>4h): Above + active_stacks context + HANDOFF.log tail 20 events

## File-Level Plan

## Tier Logic
```
gap = now() - HANDOFF.json.meta.last_written

if gap < 30 minutes:
  QUICK restore:
  - Read HANDOFF.json (position, control_point)
  - Read PHASE-STATE.md (current task)
  - Resume from position.sub_task if mid-task
  - No context reconstruction needed

elif gap < 4 hours:
  STANDARD restore:
  - Batch read: HANDOFF.json + PHASE-STATE.md + current TASK.md
  - Display context summary: "Resuming {task_id} in {phase}. Last: {last_decision}"
  - Resume

else (>4 hours):
  FULL restore:
  - Batch read: HANDOFF.json + PHASE-STATE.md + current TASK.md
  - Load active_stacks: read each stack in context.active_stacks
  - Read HANDOFF.log tail (last 20 events) for recent history
  - Display full context summary
  - Ask user: "Ready to resume?" before continuing
```

## Acceptance Criteria
- [ ] Tier detection based on gap calculation (not hardcoded tier)
- [ ] Quick restore: reads only 2 files
- [ ] Standard restore: reads 3 files in 1 batch
- [ ] Full restore: reads HANDOFF.log tail + active_stacks context
- [ ] Full restore: asks user confirmation before resuming
- [ ] Correct HANDOFF.json.meta.last_written timestamp parsing
- [ ] Works correctly even when HANDOFF.log not present (no rotation yet)

## Do / Don't
### Do
- Calculate gap từ meta.last_written (not file mtime — consistent across systems)
- Full restore: user confirmation before starting — context reset is disorienting

### Don't
- Không load tất cả files cho quick restore — defeats the purpose
- Không fail nếu HANDOFF.log missing (first session có thể chưa có log)
