Phase 1 = LLM researches the target codebase; Phase 2 = LLM writes agent files directly
using Write tool (content fully determined by research, not templates).

```
/vp-qa --focus perf   → bias research toward performance domains
/vp-qa --target <id>  → override adapter detection
```

**Core process (LLM-driven, no templates):**

```
### Phase 1: Research (REQUIRED — do not skip)

Before writing any agent file, understand the project:

1. Read .viepilot/PROJECT-META.md, STACKS.md, PROJECT-CONTEXT.md (silent if missing)
2. Detect stack from: package.json / pom.xml / go.mod / requirements.txt / Gemfile
3. List backend directory structure: find src/ app/ lib/ services/ api/ routes/ (whichever exist)
4. Read 5-10 representative source files from backend dirs
5. Count file sizes, service count, DB layer presence
6. Read .viepilot/requests/ to list known existing issues (avoid duplicate reporting)
7. Also read agents/qa-templates/rules/{stack}.md (from lib/) for stack-specific patterns

Build research summary:
  - projectName, stack, stackVersion
  - backendDirs[] (actual dirs found)
  - detectedPatterns[] (what anti-patterns were already spotted in sampling)
  - recommendedDomains[] (which scan areas are relevant given the project)
  - recommendedAgentCount (2 for small, 4-5 for large/complex)
  - knownIssues[] (from existing .viepilot/requests/)

### Phase 2: Determine output location

Use lib/qa-router.cjs to resolve adapter-specific output paths:
  - claude-code  → .claude/agents/
  - cursor-agent → .cursor/rules/
  - codex        → AGENTS.md (single file, append mode)
  - antigravity  → .agents/skills/
  - copilot      → .github/agents/

### Phase 3: Generate agent files (LLM writes directly)

Based on research summary, generate agent content and write using Write tool.

For **claude-code** (multi-agent):
- Write qa-orchestrator.md — tailored to THIS project's structure, stack, and patterns found
- Write qa-{domain}-scanner.md for each recommended domain
- Each file has correct YAML frontmatter (name, description, model, maxTurns, tools)
- Content references the ACTUAL backend dirs found, ACTUAL patterns to look for
- Orchestrator knows to create .viepilot/requests/BUG-{N}.md for found issues
- Orchestrator uses AskUserQuestion for accept/decline

For **other adapters** (single-file):
- Write one combined file with all domain instructions
- Sequential scanning procedure using that adapter's shell tools
- Same vp-request output format

Content should NOT be generic — it should reference specific dirs, patterns, and concerns
that were found during Phase 1 research.

### Phase 4: AskUserQuestion (claude-code only)

After writing files, show what was generated:
"Generated {N} agent files in {output_dir}"

AskUserQuestion:
- "Run QA scan now" → invoke qa-orchestrator immediately
- "Done for now" → exit

Text fallback: print paths + "Invoke qa-orchestrator to start the scan."
```

**Generated qa-orchestrator behavior (in target project, when user runs it):**
```
When qa-orchestrator is run:
1. Fan out to specialist subagents (claude-code) or scan sequentially (others)
2. Collect issue reports from each domain
3. Group by severity
4. For critical/high: AskUserQuestion per group (accept → create vp-request, decline → skip)
5. For medium/low: one batch confirm
6. Create .viepilot/requests/BUG-{N}.md for accepted issues
7. AskUserQuestion: "N issues logged. Run /vp-evolve to plan fixes?"
```
